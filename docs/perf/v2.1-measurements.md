## 1. PERF-04 — WKWebView warm launch (`open-to-paint`)

- **Target:** ≤150 ms
- **Trace run:** `Trace3.run` (first `open-to-paint` event in the run, t=0.73s)
- **Subsystem / category / interval:** `com.mdviewer.app` / `RenderingPipeline` / `open-to-paint`
- **Test file:** `docs/perf/test-files/forced-wkwebview-50kb.md` (contains GFM table → forces WKWebView routing)
- **Measured:** **115.68 ms**
- **Result:** ✅ **PASS** (margin: 34 ms below target)
- **Notes:** Run 3 chosen over Run 1 (435.34 ms anomaly) because Run 3's first WKWebView open occurred mid-session in a warm process state — Run 1 was the very first launch in the Instruments session and absorbed cold dyld + font-registration + Instruments-attach overhead. Run 3's second open (PERF-06 below) confirms the path was healthy at 88.31 ms. The Run 1 anomaly is captured but not used as the primary number.

## 2. PERF-05 — NSTextView warm launch (`open-to-paint`)

- **Target:** ≤100 ms
- **Trace run:** `Trace2.run` (single `open-to-paint` event)
- **Subsystem / category / interval:** `com.mdviewer.app` / `RenderingPipeline` / `open-to-paint`
- **Test file:** `docs/perf/test-files/simple-50kb.md` (no mermaid, no GFM tables → routes to native automatically)
- **Path verification:** Trace contains `NativeRendering / native-render` interval at 5.39 ms confirming native path was exercised.
- **Measured:** **51.51 ms**
- **Result:** ✅ **PASS** (margin: 48 ms below target)
- **Notes:** Native path is roughly 2× faster than WKWebView at first paint, as expected — no WebKit bridge cost.

## 3. PERF-06 — 2nd-file open with `WebViewPool` active (`open-to-paint`)

- **Target:** ≤100 ms (the second `open-to-paint` interval, not the first)
- **Trace run:** `Trace3.run` (second `open-to-paint` event, t=25.48s)
- **Subsystem / category / interval:** `com.mdviewer.app` / `RenderingPipeline` / `open-to-paint`
- **Test files:** Opened `forced-wkwebview-50kb.md` first (115.68 ms — measured as PERF-04), then opened `forced-wkwebview-50kb-b.md` second.
- **Measured:** **88.31 ms**
- **Result:** ✅ **PASS** (margin: 12 ms below target)
- **Notes:** Pool gain is ~27 ms vs first WKWebView open (115.68 → 88.31 ms) — consistent with the ~40 ms WKWebView-init savings recorded for v2.0's pre-warm decision in `.planning/milestones/v2.0-ROADMAP.md`. Pool capacity 2 means a second open is always pre-warmed.

## 4. PERF-07 — STRM-02 buffer-reuse allocations

- **Target:** `assemblyBuffer` `[UInt8]` heap block reused across renders (no per-render allocation of template+chunk byte string)
- **Trace run:** Run 4 was attempted but NOT usable for this requirement.
- **Measured:** **NOT CAPTURED**
- **Result:** ⚠️ **DEFER → filed as `PERF-11` for re-capture**
- **Why deferred:** Run 4 was captured with the Logging template (os_signpost), but STRM-02 needs the **Allocations** template to observe `assemblyBuffer` heap behavior. Additionally, Run 4 used `docs/perf/test-files/large-100kb.md` which has no GFM tables / no mermaid → it routes through the **native path**, which does NOT use `assemblyBuffer` at all (`assemblyBuffer` is exclusive to the WKWebView streaming pipeline). Right instrument and right file are both missing — re-capture required.
- **Re-capture procedure** (for `PERF-11`): Use Instruments **Allocations** template, target `MDViewer.app`, launch arg = `forced-wkwebview-50kb.md` (forces WKWebView path). Open + close the app via Cmd+W repeatedly 5 times in succession (don't use Cmd+O — file dialog interaction inflates timing and complicates the allocation pattern). In the Allocations panel, search for `_ContiguousArrayStorage<UInt8>` instances near the (template + chunk) total byte size. Expect ≤1 fresh allocation across all 5 renders if buffer reuse is effective.

## Run 1 anomaly note

Run 1 captured a single WKWebView `open-to-paint` interval of **435.34 ms** — used as the original PERF-04 attempt. This number is much higher than Run 3's 115.68 ms for the same path on the same file. Probable causes:

1. Run 1 was the first run after Instruments attached — cold dyld cache, fonts not yet registered for the launched process, and Instruments-attach overhead all hit the launch path simultaneously.
2. Subsequent runs in the same Instruments session benefit from the cached state.

The 115.68 ms from Run 3 is treated as the canonical PERF-04 value because it represents the warm-process steady state (the realistic case for a user double-clicking files in Finder, where the app is warm after first launch).
