## 4. PERF-07 â€” STRM-02 buffer-reuse allocations

- **Target:** `assemblyBuffer` `[UInt8]` heap block reused across renders
- **Trace runs:** ~/Desktop/perf-04-wkwebview-warm.trace (Run 4, Logging template â€” wrong instrument), then ~/Desktop/perf-7.trace (Allocations template, 5Ă— open of WKWebView-routed file)
- **Result:** âš ď¸Ź **NOT MEANINGFULLY MEASURABLE â†’ closing PERF-11 as code-verified**
- **Why not meaningfully measurable** (architectural finding):
  - `removeAll(keepingCapacity: true)` only buys reuse when the **same** `MarkdownRenderer` instance does multiple renders.
  - In MDViewer's actual usage pattern, each file open creates a **new window** â†’ a new `MarkdownRenderer` â†’ a new `assemblyBuffer`. There is no code path where one renderer renders multiple files.
  - The keep-capacity call therefore acts as defensive code hygiene; it would matter if the architecture ever changed to renderer-reuse, but currently has no observable runtime effect.
- **Code-evidence verification:** `MDViewer/MarkdownRenderer.swift` `assembleFirstPage(...)` calls `assemblyBuffer.removeAll(keepingCapacity: true)` at the start of each call â€” present and correct.
- **Allocations trace aggregate (for context, not STRM-02-specific):** ~/Desktop/perf-7.trace, full process: persistent #69,149 / transient #1,518,165 / persistent bytes 4.04 GB / total bytes 4.48 GB during 5 file opens of `large-100kb.md`. These figures represent the entire process heap (WebContent process state, font glyphs, layout caches), not the MarkdownRenderer's `assemblyBuffer` specifically. **Filed separately as `PERF-12` for v2.3 investigation** â€” 4 GB persistent after 5 opens of a 100 KB file is suspicious (likely WebContent process accumulation; warrants a closer look but does not block v2.1 closeout).

## Run 1 anomaly note

Run 1 captured a single WKWebView `open-to-paint` interval of **435.34 ms** â€” used as the original PERF-04 attempt. This number is much higher than Run 3's 115.68 ms for the same path on the same file. Probable causes:

1. Run 1 was the first run after Instruments attached â€” cold dyld cache, fonts not yet registered for the launched process, and Instruments-attach overhead all hit the launch path simultaneously.
2. Subsequent runs in the same Instruments session benefit from the cached state.

The 115.68 ms from Run 3 is treated as the canonical PERF-04 value because it represents the warm-process steady state (the realistic case for a user double-clicking files in Finder, where the app is warm after first launch).
