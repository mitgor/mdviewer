2. **Replace the entire `loadAndInitMermaid()` method** (lines 208-222) with:

```swift
/// Load mermaid.min.js via script-src injection — WebKit loads directly from disk,
/// bypassing the multi-megabyte IPC bridge call that evaluateJavaScript would require.
private func loadAndInitMermaid() {
    let js = """
    (function() {
        var s = document.createElement('script');
        s.src = 'mermaid.min.js';
        s.onload = function() { window.initMermaid(); };
        s.onerror = function() {
            var els = document.querySelectorAll('.mermaid-placeholder');
            els.forEach(function(el) {
                var pre = document.createElement('pre');
                var code = document.createElement('code');
                code.textContent = el.getAttribute('data-mermaid-source') || 'Mermaid failed to load';
                pre.appendChild(code);
                el.replaceWith(pre);
            });
        };
        document.head.appendChild(s);
    })()
    """
    webView.evaluateJavaScript(js)
}
```

Key points:
- The `evaluateJavaScript` call now sends ~350 bytes (the DOM injection snippet), NOT the 3MB mermaid source.
- `s.src = 'mermaid.min.js'` resolves against baseURL (Bundle.main.resourceURL), already set in `loadContent` on line 79-80.
- `s.onload` calls `window.initMermaid()` after the script has loaded and executed.
- `s.onerror` provides graceful fallback: replaces mermaid placeholders with raw source code if loading fails.
- No static cache is needed since each WKWebView loads from disk independently (WebKit handles process-level caching).

Do NOT use `callAsyncJavaScript` here — this is fire-and-forget DOM manipulation with no user data arguments to escape.
