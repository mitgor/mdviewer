# Plan 10-01: v2.1 Quality Closeout — Instruments Measurement & UAT Sign-off

## Goal

Capture the four outstanding v2.1 Instruments measurements (WKWebView warm launch, NSTextView warm launch, 2nd-file pool open, STRM-02 buffer-reuse allocations), walk through the three pending HUMAN-UAT scenarios for Phases 07 and 08, and reconcile the four `human_needed` VERIFICATION reports for Phases 06–09 — escalating any miss as a new requirement rather than fixing it inside this phase.

## Background

v2.1 (Phases 06–09) shipped on 2026-04-18 with five **acknowledged-deferred** items captured in `.planning/milestones/v2.1-ROADMAP.md` under "Issues Deferred":

1. Sub-150ms WKWebView warm launch (PERF-01) — only the v2.0 baseline (184.50ms on M4 Max) has been measured; v2.1 added pool + streaming + native paths but no post-v2.1 Instruments capture exists.
2. Sub-100ms NSTextView warm launch (PERF-02) — instrumented but never measured.
3. Sub-100ms 2nd-file pool open (PERF-03) — instrumented but never measured.
4. STRM-02 buffer-reuse effectiveness — `removeAll(keepingCapacity: true)` is in source, but heap behavior under Instruments Allocations is unconfirmed.
5. Three HUMAN-UAT scenarios pending across Phases 07 and 08; four VERIFICATION reports stuck at `status: human_needed`.

Per v2.2 PITFALLS.md #16, `xctrace` cannot enable signposts headlessly from CLI — these measurements must be captured via the Instruments GUI on a developer Mac and committed to the repo as a manual record. CI gating of perf is explicitly out of scope for v2.2 (see REQUIREMENTS.md "Out of Scope" table).

This phase is the closeout pass that converts those deferrals into either signed-off measurements or filed follow-up requirements before v2.2's CI/Sparkle/Homebrew work begins.

## Approach

**Measurement methodology** (per PITFALLS.md #16, RESEARCH §6):

- Use the Instruments **Logging template** (or a built-from-GUI `os_signpost` template) to capture `OSSignposter` intervals from the `com.mdviewer.app` subsystem, `RenderingPipeline` category. The relevant interval names already wired in v2.1 are: `open-to-paint`, `file-read`, `parse-feed`, `stream-first-chunk`.
- Use the Instruments **Allocations** instrument for STRM-02 — filter to the `com.mdviewer.app` process and watch the `assemblyBuffer` `[UInt8]` backing store across repeated file opens.
- Run on the developer Mac. Record the M-series chip and macOS version in the measurement file (e.g. "M4 Max, macOS 15.4").
- For warm launches, open and close the app once before measuring so the bundle, dyld cache, and font registration are hot.
- Use a representative ~50KB simple markdown file (no mermaid, no GFM tables) for PERF-01, PERF-02, and PERF-03; a larger ~100KB file for PERF-07 buffer-reuse.

**Sign-off methodology:**

- For each pending UAT scenario, perform the documented test in the running app, then update the `*-HUMAN-UAT.md` `result:` line with `PASS` or `FAIL: <one-line reason>` and a dated note.
- For each VERIFICATION report at `human_needed`, after measurement/UAT is complete, update the frontmatter `status:` to `verified` (or `verified_with_followup` if a new REQ was filed) and append a "Closeout Note" section pointing to the measurement entry or new REQ.

**Escalation rule:**

- If WKWebView warm launch >150ms: file `PERF-08` in `.planning/REQUIREMENTS.md` ("WKWebView warm launch regression — measured Xms vs 150ms target, follow-up phase needed"). Do NOT attempt to fix inside Phase 10.
- If NSTextView warm launch >100ms: file `PERF-09` similarly.
- If 2nd-file pool open >100ms: file `PERF-10` similarly.
- If buffer-reuse trace shows per-render allocations: file `PERF-11` ("STRM-02 buffer reuse not effective in practice — heap allocations observed per render").
- If a UAT scenario fails for non-perf reasons: file a new functional requirement (e.g. `BUG-V21-XX`) describing the observed regression.

### Task 3 — Capture PERF-04: WKWebView warm launch (20 min)

- [ ] Open Instruments (`Xcode → Open Developer Tool → Instruments`), choose the **Logging** template (or any custom template with the **os_signpost** instrument; build one once via the GUI per RESEARCH §6 if no suitable template ships).
- [ ] Set the target to the built `MDViewer.app`. Add a launch argument pointing at `simple-50kb.md` (so the file opens via `argv[1]` path in `applicationDidFinishLaunching`).
- [ ] **Warm the system:** launch the app once via Finder double-click, close it. This populates dyld cache, registers fonts, etc.
- [ ] Press Record in Instruments. The app launches and renders the file. Stop recording after the WKWebView paints the first chunk (visible as content on screen).
- [ ] Filter the os_signpost track to subsystem `com.mdviewer.app`, category `RenderingPipeline`. Locate the `open-to-paint` interval. Read its duration.
- [ ] **Force the WKWebView path:** if `simple-50kb.md` routes through the native path (because it lacks mermaid/tables), use the **View → Toggle Native/Web Rendering** menu to switch to WKWebView before measuring, OR use a markdown file that contains a GFM table or mermaid block to force WKWebView routing.
- [ ] Save the trace as `perf/v2.1-perf-04-wkwebview-warm.trace` (or similar) — do NOT commit the binary `.trace` to git, but record its filename in the measurement file.
- [ ] Update entry #1 in `docs/perf/v2.1-measurements.md` with the measured ms value and PASS/FAIL.
- [ ] If FAIL (>150ms): file `PERF-08` in `.planning/REQUIREMENTS.md` (see Task 7 for format).

### Task 4 — Capture PERF-05: NSTextView warm launch (15 min)

- [ ] Read the `open-to-paint` interval duration.
- [ ] Update entry #2 with measured value and PASS/FAIL.

### Task 5 — Capture PERF-06: 2nd-file pool open (15 min)

- [ ] Confirm the second open used a pre-warmed view from `WebViewPool` (not a fresh `WebContentView` fallback) — look for replenish-related signposts if present, or simply trust that with capacity 2 and only one prior dequeue, the pool is non-empty.
- [ ] Update entry #3 with measured value and PASS/FAIL.
- [ ] If FAIL (>100ms): file `PERF-10`.

### Task 6 — Capture PERF-07: STRM-02 buffer-reuse allocations (30 min)

- [ ] Open Instruments, choose the **Allocations** template (single instrument: Allocations).
- [ ] Target the built `MDViewer.app`. Record while opening `large-100kb.md` **at least 5 times in succession** (close window each time, reopen via Cmd+O).
- [ ] Filter Allocations by category to the `assemblyBuffer` `[UInt8]` storage — search the allocation list for repeated allocations matching the template+chunk byte size (~100KB+template overhead). The Swift type name shown will be related to `_ContiguousArrayStorage<UInt8>`.
- [ ] Count net new allocations of that size per render cycle. Ideal: 1 allocation total (or near-zero growth) across all 5 opens.
- [ ] Save the trace; record its name in the measurement file.
- [ ] Update entry #4 with the observed allocation behavior. PASS = stable allocation count across renders; FAIL = repeated per-render allocations.
- [ ] If FAIL: file `PERF-11`.
