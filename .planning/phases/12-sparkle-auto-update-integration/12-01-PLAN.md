## Goal

Make Sparkle 2.9.1+ a first-class dependency of the MDViewer app — resolved through SPM, linked into the app bundle, configured via Info.plist, owned by a long-lived AppDelegate property, and user-reachable through a "Check for Updates…" menu item. After this plan ships, the client side of the auto-update system is complete; the only remaining unknowns are the real public key (Plan 3) and the appcast URL's 200-vs-404 response (Plan 2).

## Background

- MDViewer currently has **zero SPM dependencies**. cmark-gfm is fully vendored under `Vendor/cmark-gfm/`. There is no `Package.swift`, no `Package.resolved`, no `XCRemoteSwiftPackageReference` entry in `project.pbxproj`. Sparkle will be the first. This means "add an SPM dep" is a multi-step task (manifest → project.yml → xcodegen → resolve → commit lockfile), not a one-liner.
- XcodeGen's project-spec `packages:` block is the canonical way to declare an SPM dep in a project-yml-driven repo. It maps to `XCRemoteSwiftPackageReference` + `XCSwiftPackageProductDependency` entries in the generated pbxproj.
- `Scripts/build_and_sign.sh` (Phase 11) already runs `xcodegen generate` on every CI build, so the CI side will pick up any project.yml change automatically — but the pbxproj AND the new `Package.resolved` must still be committed so the SPM cache key in `release.yml` (`hashFiles('Package.resolved', ...swiftpm/Package.resolved')`) works on first run.
- Sparkle 2.9.1 (released 2026-03-29) is the current stable release. It explicitly added "Xcode 26.4 beta compiler warnings" fixes, so Xcode 26.2 on the `macos-26` runner is fully supported. Pin conservatively with `.upToNextMajor(from: "2.9.1")` — we want patch-level updates within 2.x but not a surprise 3.0.
- `SPUStandardUpdaterController` is Sparkle's turnkey controller for apps without a preferences window — which is exactly MDViewer (see `.planning/PROJECT.md` Out of Scope: "Preferences window — no settings to persist"). It owns an `SPUUpdater` internally and exposes `checkForUpdates(_:)` as an `@IBAction` that any NSMenuItem can target directly. Init parameters: `startingUpdater: true` + both delegates `nil`. CRITICAL: it must be held as a stored property on AppDelegate; if deallocated, the auto-check timer dies silently.
- Info.plist additions are declarative — no code reads these values, Sparkle does, on `SPUUpdater` init. The five keys are documented in Sparkle's customization reference. `SUEnableSystemProfiling=false` is explicit (the default is also false, but PROJECT.md's privacy posture makes it worth writing down).
- The `SUPublicEDKey` that ships in this plan is a **PLACEHOLDER** — a 44-character literal that is syntactically a base64 Ed25519 public key but has no matching private key in existence. Plan 3 replaces it in a dedicated single-line commit after the operator has run `generate_keys` on the developer Mac. Shipping the placeholder is safe because (a) Plan 2's CI won't produce a signed appcast until Plan 3's private key is in `SPARKLE_ED_PRIVATE_KEY`, (b) the placeholder key has no matching private key so NO signature could ever verify against it, and (c) Phase 12 does NOT tag a v2.2.0 release — that happens after Plan 3.
- Menu placement: under the app menu, between "About MDViewer" and the separator-then-Quit block. This is the Firefox/Office/Sparkle-community convention. Apple's HIG doesn't mandate a location. No keyboard shortcut (leave `keyEquivalent: ""`) — Sparkle intentionally doesn't claim one.

**Locked decisions (apply verbatim):**
- Sparkle version pin: `.upToNextMajor(from: "2.9.1")` via Package.swift; project.yml uses `from: "2.9.1"` (XcodeGen's shorthand for the same constraint).
- Package URL: `https://github.com/sparkle-project/Sparkle` (canonical; HTTPS; no trailing slash).
- Info.plist `SUFeedURL`: `https://github.com/mitgor/mdviewer/releases/latest/download/appcast.xml` (phase-context-specified; Pitfall #6 HTTPS-only is satisfied).
- Info.plist `SUPublicEDKey` placeholder: `PHASE12PLACEHOLDER0000000000000000000000000=` (44 chars, base64-alphabet-valid, ends with `=` padding — Sparkle's format validator accepts the structure, but no private key matches). Plan 3 Task N replaces this with the real key in one focused commit.
- Info.plist `SUScheduledCheckInterval`: `<integer>86400</integer>` (24h).
- Info.plist `SUEnableAutomaticChecks`: `<true/>`.
- Info.plist `SUEnableSystemProfiling`: `<false/>` (explicit, per PROJECT.md privacy posture).
- AppDelegate property visibility: `private let updaterController` — match the existing access-control style in the file; nothing outside AppDelegate needs the instance.
- Menu item title: `"Check for Updates…"` (ellipsis is the Unicode `…`, not three dots — matches existing "Open…" convention in the File menu).

## Approach

Edit four logical surfaces in order, each as its own atomic commit so a reverter can bisect cleanly:

1. **Introduce SPM dependency wire-up.** Create `Package.swift` (minimal manifest naming Sparkle — this is what XcodeGen and `xcodebuild -resolvePackageDependencies` both honour). Edit `project.yml` to add a top-level `packages:` block and append `- package: Sparkle` under `targets.MDViewer.dependencies`. Do NOT hand-edit `project.pbxproj` — XcodeGen regenerates it. Note: XcodeGen preserves package refs in the regenerated pbxproj correctly as long as they're declared in project.yml.

2. **Regenerate + resolve + commit.** Run `xcodegen generate`, then `xcodebuild -project MDViewer.xcodeproj -scheme MDViewer -resolvePackageDependencies`. This writes `MDViewer.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved`. Commit all three of: updated `project.pbxproj`, updated `project.yml`, new `Package.resolved`, and new `Package.swift`. Verify no secondary changes to the cmark-gfm targets slipped in.

3. **Info.plist additions.** Use `PlistBuddy -c "Add …"` idempotently for each of the five keys. The placeholder `SUPublicEDKey` value is a constant defined in this plan (§Background); Plan 3 will `PlistBuddy -c "Set :SUPublicEDKey …"` to replace it.

4. **AppDelegate wiring.** Add `import Sparkle` to the top of `AppDelegate.swift`. Add `private let updaterController = SPUStandardUpdaterController(startingUpdater: true, updaterDelegate: nil, userDriverDelegate: nil)` as a stored property near the existing `renderer` / `template` properties. In `setupMenu()`, insert a new `NSMenuItem(title: "Check for Updates…", action: #selector(SPUStandardUpdaterController.checkForUpdates(_:)), keyEquivalent: "")` after the "About MDViewer" item, setting its `.target = updaterController` (the selector lives on the controller, not on AppDelegate — this is the standard pattern; `NSMenuItem` without an explicit target walks the responder chain, but Sparkle's method lives OFF the chain so we must set target explicitly).

**Why split this way:** Each of the four tasks touches a distinct file surface (or pair of surfaces) and has a distinct verification command. Merging them would produce a single commit where a build failure couldn't be attributed to a specific change. Keeping them separate lets `git bisect` do its job if a regression surfaces in Plan 2's CI integration.
