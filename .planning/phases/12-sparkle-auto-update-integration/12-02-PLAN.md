## Style conventions for Scripts/*.sh (from Phase 11 scripts — match these)

- First line: `#!/usr/bin/env bash`
- Second line: `# Usage: <script> <positional args>` + one-sentence purpose
- Third+ lines: additional context / references to pitfalls / what the script does NOT do
- `set -euo pipefail` before any logic
- Positional args use `${1:?arg-name required}` gating
- Env-var expectations use `: "${VAR_NAME:?must be set}"`
- `{ echo "::error::…"; exit 1; }` for failures (GitHub Actions annotation)
- `echo "PASS …"` or `echo "<action> …"` for success notes
- No Python heredocs unless unavoidable; prefer awk/sed/grep. Exception: XML post-processing legitimately warrants Python 3 (stdlib `xml.etree.ElementTree` is preinstalled on macos-26).
- Never use `echo "$SECRET"` — use `printf '%s' "$SECRET"` to avoid newline-addition on some shells. Even better: pipe directly to stdin of the consuming tool.

## Sparkle generate_appcast CLI (Sparkle 2.9 — from --help):

```
generate_appcast [options] <staging-directory>

  -f <key-file>, --ed-key-file <path>
        Path to a file containing the EdDSA private key. Use `-` for stdin.
  -o <path>
        Output appcast.xml path. Default: <staging>/appcast.xml.
  --link <url>
        The appcast channel <link> tag — typically the project homepage.
  --download-url-prefix <url>
        URL prefix prepended to DMG filenames to form enclosure URLs.
  --maximum-versions <n>
        Keep only the N most recent items in the output appcast.
  --maximum-deltas <n>
        Deltas per version (we DON'T use deltas; set to 0).
```

Notes:
- `generate_appcast` scans <staging-directory> for `.dmg`, `.zip`, `.pkg`, `.tar.*` and creates one `<item>` per file.
- It reads <sparkle:version> and <sparkle:shortVersionString> from the DMG's bundled Info.plist (the .app inside).
- It computes the enclosure `length="…"` from the DMG size and `sparkle:edSignature="…"` from Ed25519 over the DMG bytes. No separate sign_update call is needed.

## Goal

Make `.github/workflows/release.yml` produce, on every `v*` tag push, a draft GitHub Release whose assets include both the notarized stapled `MDViewer-<version>.dmg` and an EdDSA-signed `appcast.xml` that conforms to MDViewer's rollout and metadata conventions. Do this without leaking the Ed25519 private key to disk, without hardcoding the Sparkle binary path, and without regressing the Phase 11 build-and-sign flow.

## Background

- Phase 11's pipeline finishes at step 14 (dSYM upload) + step 15 (draft release upload). Step 12.5 (generate_appcast) logically belongs between step 12 (verify_release) and step 13 (notarytool log upload), because generate_appcast reads the DMG post-staple to compute the enclosure's length and Ed25519 signature — if the DMG is modified after signature computation, the signature becomes invalid. Research §Architecture diagram confirms this placement.
- Step 10.5 (verify_bundle) MUST happen after `make_dmg.sh` (step 10) but before `notarize.sh` (step 11), because notarize.sh staples to the DMG and any bundle-layout problem is cheaper to catch pre-staple. verify_release.sh (step 12) does post-staple sanity checks but does not look at bundle composition.
- The Ed25519 private key is delivered to the runner as `SPARKLE_ED_PRIVATE_KEY` secret. Pitfall B (Research) documents the risk of newline introduction during secret round-tripping; the defence is to pipe the raw secret through `generate_appcast --ed-key-file -` stdin. Stdin avoids tempfiles, avoids `ps`-visible args, and avoids the newline-on-echo concern (piped stdin preserves bytes verbatim through bash).
- `generate_appcast`'s path is under `DerivedData/<hash>/SourcePackages/artifacts/sparkle/Sparkle/bin/`. The `<hash>` changes per build on hosted runners (Pitfall A). The script must `find` it. Plan 1 adds the Sparkle SPM dependency, and Phase 11's build_and_sign.sh runs `xcodebuild archive` which triggers SPM resolution — so by the time step 12.5 runs, the binary is on disk. Fail-loud if it isn't.
- `generate_appcast` does not itself set `phasedRolloutInterval`, `releaseNotesLink`, or `minimumSystemVersion`. It emits the base `<item>` with `<enclosure>`, `<sparkle:version>`, and `<sparkle:shortVersionString>`. The other three are injected by the post-processor. `minimumSystemVersion` does have a `generate_appcast` CLI flag in recent Sparkle versions, but post-processing gives us one code path to maintain for all three injected fields — preferred over mixing CLI flags and post-processing.
- Decision on single-DMG staging vs. full-history regeneration (from Research §6): This plan uses **single-DMG staging** — `generate_appcast` produces an appcast with exactly one `<item>` for the just-built version. Sparkle clients read the latest `<item>` and the GitHub Releases page is the historical record; there is no requirement to carry old items in the appcast. Rationale: (a) simpler (no artefact to download from the previous release), (b) the primary `/releases/latest/download/appcast.xml` URL only matters for "latest" anyway, (c) matches the research recommendation, (d) keeps the workflow self-contained (no inter-release dependency). If this ever bites us in the future, switching to regenerate-from-history is a one-script change; the append-to-previous variant is explicitly rejected because it re-introduces inter-release ordering bugs. Document this decision in the script header.
- Post-processing tool choice: Python 3 stdlib `xml.etree.ElementTree` (preinstalled on macos-26, verified in Research §Environment Availability). Alternative `xmlstarlet` requires a brew install and is slower. `sed` / `grep` are insufficient for namespace-aware XML; using them risks corrupting the `sparkle:` prefix handling.
- `verify_bundle.sh`'s checks go beyond SPK-10's literal "exactly one Sparkle.framework" — the research's §Runtime State Verification section offers seven checks that are all cheap and catch different regression classes. We include all seven: (1) framework count = 1, (2) framework at standard path, (3) Installer.xpc present, (4) Downloader.xpc present, (5) consistent team ID across app+framework+XPCs (Pitfall #15 — mismatched team IDs cause XPC rejection at runtime), (6) hardened runtime on app + framework, (7) required Sparkle Info.plist keys present. We add an 8th: (8) CFBundleVersion is a pure integer (guards against a Plan 3 regression). The script is small (~80 lines) and makes regressions loud.

**Locked decisions (apply verbatim):**
- verify_bundle.sh lives at `Scripts/verify_bundle.sh` and takes two args: `<app-path> <team-id>` (match the existing `build_and_sign.sh` two-arg convention, e.g. `./Scripts/verify_bundle.sh build/export/MDViewer.app V7YK72YLFF`).
- generate_appcast.sh lives at `Scripts/generate_appcast.sh` and takes two args: `<version> <dmg-path>`.
- Appcast is output at `build/appcast.xml`.
- Post-processor is inline Python 3 heredoc (not a separate `.py` file — 30 lines; breaking it out adds a file with no reuse).
- Channel `<link>` in the appcast points at `https://github.com/mitgor/mdviewer` (the repo landing page). Per-item `<sparkle:releaseNotesLink>` points at the tag page `https://github.com/mitgor/mdviewer/releases/tag/v${VERSION}`.
- Single-DMG staging (as described above). Not full-history regeneration. Not append-to-previous.
- Python 3 for post-processing (not xmlstarlet; not sed).
- stdin for key materialisation (not tempfile). Tempfile-with-trap pattern is documented as the fallback in a script comment in case a future Sparkle version removes stdin support, but is NOT the active code path.
- `--timestamp` / secure-timestamp concerns do NOT apply here (that's codesign territory, not generate_appcast). `--maximum-deltas 0` passed explicitly (MDViewer does not use delta updates per the Out-of-Scope list).
- action-gh-release `files:` list: append `build/appcast.xml` AFTER the existing DMG entry; `fail_on_unmatched_files: true` stays true so a missing appcast.xml breaks CI.

## Approach

Three atomic tasks, each one commit:

1. **Task 1: `Scripts/verify_bundle.sh` new file.** A self-contained shell script, ~80 lines, that performs eight invariants. Pattern matches `verify_release.sh` exactly for style/shape.

2. **Task 2: `Scripts/generate_appcast.sh` new file.** A self-contained shell script, ~100 lines, that discovers the Sparkle binary, stages the DMG, pipes the Ed25519 private key via stdin to `generate_appcast`, post-processes the resulting XML with a Python 3 heredoc, and deposits `build/appcast.xml`. Signs loudly when its inputs are wrong.

3. **Task 3: wire them into `.github/workflows/release.yml`.** Two new steps inserted into the existing single job, plus `build/appcast.xml` added to the gh-release step's `files:` list. No restructure of the job — just surgical edits.

Each task is its own commit so the history bisects cleanly. Task 1 and Task 2 can technically be written in either order (they're independent), but commits should land in the order: script → script → workflow, so the workflow commit cleanly shows the new steps consuming scripts that already exist in the working tree.

```bash
#!/usr/bin/env bash
# Usage: verify_bundle.sh <app-path> <team-id>
# Pre-notarize bundle-layout smoke test for Phase 12 (Sparkle integration).
# Guards against: SPM double-bundle regression (sparkle-project/Sparkle#1689),
# mismatched team IDs on XPC services (Pitfall #15 — causes runtime XPC
# rejection with no user-visible error), missing XPC services, unstripped
# hardened-runtime, and Plan-3 CFBundleVersion-string regression.
#
# Runs AFTER Scripts/make_dmg.sh (so the .app is signed and the framework
# is embedded) and BEFORE Scripts/notarize.sh (cheaper to fail here than
# after paying the ~5min notarize wait).
set -euo pipefail

APP="${1:?app bundle path required (e.g. build/export/MDViewer.app)}"
TEAM_ID="${2:?team id required (e.g. V7YK72YLFF)}"

[ -d "$APP" ] || { echo "::error::$APP not found or not a directory"; exit 1; }

# 1. Exactly one Sparkle.framework — guards SPM double-bundle (SPK-10).
COUNT=$(find "$APP" -type d -name 'Sparkle.framework' | wc -l | xargs)
if [ "$COUNT" != "1" ]; then
  echo "::error::expected exactly 1 Sparkle.framework in $APP, found $COUNT"
  find "$APP" -type d -name 'Sparkle.framework'
  exit 1
fi
echo "PASS 1x Sparkle.framework"

# 2. Framework is at the standard path.
FW="$APP/Contents/Frameworks/Sparkle.framework"
[ -d "$FW" ] || { echo "::error::Sparkle.framework not at $FW"; exit 1; }
echo "PASS Sparkle.framework at Contents/Frameworks/"

# 3+4. XPC services present inside the framework.
XPC_BASE="$FW/Versions/B/XPCServices"
for XPC in Installer Downloader; do
  [ -d "$XPC_BASE/${XPC}.xpc" ] \
    || { echo "::error::${XPC}.xpc missing at $XPC_BASE/${XPC}.xpc"; exit 1; }
done
echo "PASS Installer.xpc + Downloader.xpc present"

# 5. Consistent team ID across app + framework + XPCs (Pitfall #15).
# Mismatched team IDs on XPC services cause Sparkle 2's hardened XPC policy
# to reject the update at runtime with no user-visible error.
for CHECK in \
  "$APP" \
  "$FW" \
  "$XPC_BASE/Installer.xpc" \
  "$XPC_BASE/Downloader.xpc"; do
  codesign -dvvv "$CHECK" 2>&1 | grep -q "TeamIdentifier=${TEAM_ID}" \
    || { echo "::error::wrong team identifier on $CHECK (expected $TEAM_ID)"; exit 1; }
done
echo "PASS consistent team identifier $TEAM_ID across app+framework+xpcs"

# 6. Hardened runtime on app and framework.
for CHECK in "$APP" "$FW"; do
  codesign -dvvv "$CHECK" 2>&1 | grep -qE "flags=.*runtime" \
    || { echo "::error::hardened runtime not set on $CHECK"; exit 1; }
done
echo "PASS hardened runtime on app + framework"

# 7. Required Sparkle Info.plist keys present.
PLIST="$APP/Contents/Info.plist"
for KEY in SUFeedURL SUPublicEDKey SUEnableAutomaticChecks SUScheduledCheckInterval; do
  /usr/libexec/PlistBuddy -c "Print :$KEY" "$PLIST" >/dev/null 2>&1 \
    || { echo "::error::$PLIST missing required Sparkle key: $KEY"; exit 1; }
done
echo "PASS required Sparkle Info.plist keys present"

# 8. CFBundleVersion is a pure integer (SPK-08 gate — guards Plan 3 regression).
# Sparkle compares CFBundleVersion lexicographically; a dotted-string like
# "2.10" sorts LESS THAN "2.9" which would cause a downgrade scenario.
CFB_VER=$(/usr/libexec/PlistBuddy -c "Print :CFBundleVersion" "$PLIST")
if [[ ! "$CFB_VER" =~ ^[0-9]+$ ]]; then
  echo "::error::CFBundleVersion is not a pure integer: '$CFB_VER'"
  echo "::error::Sparkle compares CFBundleVersion lexicographically — this would break updates."
  echo "::error::See SPK-08 in .planning/REQUIREMENTS.md and Plan 12-03 migration task."
  exit 1
fi
echo "PASS CFBundleVersion=$CFB_VER (integer)"

echo "Bundle verified: $APP"
```

```bash
# Produces build/appcast.xml, EdDSA-signed by Sparkle's generate_appcast,
# with MDViewer post-processing injected per-item:
#   - <sparkle:phasedRolloutInterval>43200</sparkle:phasedRolloutInterval>  (12h, SPK-07)
#   - <sparkle:releaseNotesLink>https://github.com/mitgor/mdviewer/releases/tag/v${VERSION}</sparkle:releaseNotesLink>  (SPK-12)
#   - <sparkle:minimumSystemVersion>13.0.0</sparkle:minimumSystemVersion>  (SPK-09 appcast side)
#
# Staging strategy: SINGLE-DMG. We stage only the just-built DMG so the
# output appcast has exactly one <item>. The GitHub Releases page is the
# historical record; Sparkle clients read the latest <item> and do not
# need prior-version items in the appcast. (See Research §6 + §Planning
# Implications; alternative full-history-regeneration or append-to-previous
# strategies are explicitly rejected.)
#
# Key materialisation: the EdDSA private key is streamed through STDIN
# to `generate_appcast --ed-key-file -` and NEVER written to runner disk.
# This avoids Pitfall B (newline introduction during echo/file round-trip)
# and keeps the key out of `ps`, $RUNNER_TEMP, and job logs.
#
# Fallback (NOT active): if a future Sparkle version drops stdin support,
# the tempfile-with-trap pattern is:
#     KEYF=$(mktemp -t sparkle_ed_XXXXXX) && trap 'rm -f "$KEYF"' EXIT
#     printf '%s' "$SPARKLE_ED_PRIVATE_KEY" > "$KEYF" && chmod 600 "$KEYF"
#     "$GENERATE_APPCAST" --ed-key-file "$KEYF" …
# Leave as a comment; do not activate unless stdin is removed upstream.
set -euo pipefail

VERSION="${1:?version required (e.g. 2.2.0)}"
DMG_PATH="${2:?dmg path required (e.g. build/MDViewer-2.2.0.dmg)}"

: "${SPARKLE_ED_PRIVATE_KEY:?SPARKLE_ED_PRIVATE_KEY must be set in env}"
[ -f "$DMG_PATH" ] || { echo "::error::$DMG_PATH not found"; exit 1; }

# 1. Locate generate_appcast in DerivedData — path hash varies per-build
#    on hosted runners, so we discover it rather than hardcode.
#    (Pitfall A — hardcoded paths work locally, break in CI.)
GENERATE_APPCAST=$(find ~/Library/Developer/Xcode/DerivedData \
    -type f -name generate_appcast \
    -path '*SourcePackages/artifacts/sparkle/Sparkle/bin/*' 2>/dev/null \
    | head -n1)

if [ -z "$GENERATE_APPCAST" ]; then
  echo "::error::generate_appcast not found in DerivedData."
  echo "::error::Has Sparkle SPM been resolved? build_and_sign.sh should have"
  echo "::error::triggered resolution via 'xcodebuild archive'. Check earlier logs."
  exit 1
fi
chmod +x "$GENERATE_APPCAST"
echo "Found generate_appcast at: $GENERATE_APPCAST"

# 2. Stage the DMG alone in a directory. generate_appcast scans a
#    directory for .dmg / .zip / .pkg / .tar.* files and produces one
#    <item> per file. Single-DMG staging => single-<item> appcast.
STAGING="${RUNNER_TEMP:-/tmp}/appcast-staging"
rm -rf "$STAGING"
mkdir -p "$STAGING"
cp "$DMG_PATH" "$STAGING/"

# 3. Invoke generate_appcast with the private key streamed via stdin.
#    --maximum-deltas 0: MDViewer does not use Sparkle delta updates
#      (per milestone Out-of-Scope list).
#    --link: appcast channel <link> — project landing page.
#    --download-url-prefix: where each <item>'s <enclosure url> points.
mkdir -p build
RAW_APPCAST="${RUNNER_TEMP:-/tmp}/appcast.xml"
printf '%s' "$SPARKLE_ED_PRIVATE_KEY" \
  | "$GENERATE_APPCAST" \
      --ed-key-file - \
      --maximum-deltas 0 \
      --link "https://github.com/mitgor/mdviewer" \
      --download-url-prefix "https://github.com/mitgor/mdviewer/releases/download/v${VERSION}/" \
      -o "$RAW_APPCAST" \
      "$STAGING"

[ -s "$RAW_APPCAST" ] \
  || { echo "::error::generate_appcast produced empty appcast.xml"; exit 1; }

# 4. Post-process: inject phasedRolloutInterval + releaseNotesLink +
#    minimumSystemVersion under each <item>, idempotent (upsert: if a
#    child already exists, set its text; else create).
python3 - "$RAW_APPCAST" "$VERSION" <<'PY'
import sys
import xml.etree.ElementTree as ET

path, version = sys.argv[1], sys.argv[2]
SPARKLE_NS = 'http://www.andymatuschak.org/xml-namespaces/sparkle'
ET.register_namespace('sparkle', SPARKLE_NS)
ET.register_namespace('dc', 'http://purl.org/dc/elements/1.1/')

tree = ET.parse(path)
root = tree.getroot()

def upsert(parent, local_name, text):
    qname = '{%s}%s' % (SPARKLE_NS, local_name)
    existing = parent.find(qname)
    if existing is None:
        existing = ET.SubElement(parent, qname)
    existing.text = text

item_count = 0
for item in root.iter('item'):
    upsert(item, 'phasedRolloutInterval', '43200')
    upsert(item, 'releaseNotesLink',
           'https://github.com/mitgor/mdviewer/releases/tag/v%s' % version)
    upsert(item, 'minimumSystemVersion', '13.0.0')
    item_count += 1

if item_count == 0:
    print('::error::No <item> elements found in appcast XML — generate_appcast failed silently?')
    sys.exit(1)

tree.write(path, xml_declaration=True, encoding='utf-8')
print('Post-processed %d <item> element(s) in %s' % (item_count, path))
PY

# 5. Copy to build/ so the workflow's action-gh-release step can upload it.
cp "$RAW_APPCAST" build/appcast.xml

# 6. Sanity-check: the output has at least one <sparkle:edSignature>.
#    If this fails, the key didn't survive the pipe or generate_appcast
#    had a silent error — either way, do NOT upload an unsigned appcast.
grep -q 'sparkle:edSignature' build/appcast.xml \
  || { echo "::error::build/appcast.xml missing EdDSA signature"; exit 1; }

# 7. Sanity-check: our three post-processed elements are present.
for ELEMENT in 'phasedRolloutInterval>43200' 'minimumSystemVersion>13.0.0' "releaseNotesLink>https://github.com/mitgor/mdviewer/releases/tag/v${VERSION}"; do
  grep -q "$ELEMENT" build/appcast.xml \
    || { echo "::error::build/appcast.xml missing post-processed element: $ELEMENT"; exit 1; }
done

echo "Generated build/appcast.xml for v${VERSION}"
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Secrets ↔ runner memory | `SPARKLE_ED_PRIVATE_KEY` is injected as an env var; exits the runner when the job ends. Our code path never writes it to disk. |
| Runner ↔ Sparkle upstream SPM artifact | `generate_appcast` binary comes from the SPM-resolved Sparkle package. Binary trust is the same as Plan 1's — pinned via `Package.resolved`. |
| Runner ↔ GitHub Releases | `build/appcast.xml` is uploaded as a release asset. This is the same trust surface as the DMG — `GITHUB_TOKEN` scoped to `contents: write` by release.yml permissions. |
| Unpublished draft ↔ end users | The draft release is NOT visible to anonymous users; the `/releases/latest/download/appcast.xml` URL will not resolve until the draft is manually published (preserves the Phase 11 security boundary — T-11-14). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-07 | Information Disclosure | `SPARKLE_ED_PRIVATE_KEY` leaked via runner filesystem (Pitfall B — `echo` newline, tempfile residue) | mitigate | Key streamed via stdin (`printf '%s' … \| generate_appcast --ed-key-file -`); never written to `$RUNNER_TEMP` or any file. Tempfile fallback documented in script header as COMMENT only, not active code path. Script never invokes `set -x`; the workflow step never invokes `set -x`. |
| T-12-08 | Information Disclosure | Shell tracing (`set -x`) inadvertently enabled surfaces private key in job log | mitigate | Neither `generate_appcast.sh` nor the workflow step enables `set -x`. `set -euo pipefail` is all we set. GitHub's secret-redaction layer is a backstop, not the primary control. |
| T-12-09 | Tampering | Malicious DMG ingested from prior runs of verify_bundle.sh via shared `STAGING` dir | mitigate | `generate_appcast.sh` does `rm -rf "$STAGING"; mkdir -p "$STAGING"` before each run — no residue from a prior runner invocation (though on hosted runners this is a fresh VM anyway). |
| T-12-10 | Tampering | Appcast XML hand-modified after signing to swap enclosure URL | mitigate | `generate_appcast` signs the ENCLOSURE BYTES (SHA of DMG content), not the XML text, and the signature is emitted as an XML attribute. A URL swap doesn't break the signature per se — BUT Sparkle's download validates the Ed25519 signature over the downloaded bytes, so swapping the URL only succeeds if the attacker can produce a different DMG whose bytes are identically signed — infeasible without the private key. Residual risk: an attacker who intercepts the JSON-stream HTTPS to GitHub could substitute a different valid signed DMG from our own back-catalog (downgrade). Mitigated by SPK-08 monotonic CFBundleVersion (Plan 3) — Sparkle refuses lexicographic downgrades. |
| T-12-11 | Denial of Service | `generate_appcast` binary missing from DerivedData at step 12.5 time | mitigate | Script fails fast with `::error::generate_appcast not found. Has Sparkle SPM been resolved?` — clear diagnostic pointing at the resolve step. Since build_and_sign.sh (step 9) runs `xcodebuild archive`, which triggers resolution, the only way this fires is if step 9 catastrophically failed — in which case we already stopped. Fail-loud beats fail-silent. |
| T-12-12 | Tampering | Post-processor Python script mangles EdDSA signature attribute during write | accept → mitigate | Research A4: `xml.etree.ElementTree` preserves text nodes and attribute bytes verbatim on round-trip. The `sparkle:edSignature` is an XML attribute (not a text node) with base64 content that xml.etree treats as an opaque string. Sanity check in Task 2 step 6 (`grep -q 'sparkle:edSignature'`) catches any regression. Residual risk: a future Python stdlib release changes attribute-ordering behaviour in a way that triggers a Sparkle parser bug — deemed extremely low probability. |
| T-12-13 | Spoofing | verify_bundle.sh trusts TEAM_ID arg — a typo would pass verification against the wrong team | accept | Arg is passed from `$APPLE_TEAM_ID` env in the workflow, which comes from the `APPLE_TEAM_ID` repo secret. Mitigated by SHA-pinning (already in place — `@3bb12739c298aeb8a4eeaf626c5b8d85266b0e65`). We don't re-pin here; we inherit Plan 11-02's pin. |
