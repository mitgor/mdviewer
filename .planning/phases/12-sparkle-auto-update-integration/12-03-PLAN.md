## The integer scheme (locked — per Research Pitfall E + Assumption A3)

- `2.2.0` → `220` (strip dots)
- `2.2.1` → `221`
- `2.3.0` → `230` (> 221)
- `2.10.0` → `2100` (> 290 — so the scheme tolerates two-digit minors)
- `2.2.10` → `2210` (> 220, > 221, > 229)
- Pre-release tags (`2.2.0-rc.2`) → `220` (strip `-rc.N`; same integer as final `2.2.0`)

Rationale for strip-dots over `git rev-list --count`: simpler (one substitution), human-readable, monotonic as long as each component < 10000 (Research A3). No git-history access in CI. rc.N collisions are accepted: rc.1 and rc.2 both stamp 220, which means a machine with rc.2 installed will not auto-update to final v2.2.0 — fine because rc.N tags are pre-release and not served from `/releases/latest/`.

## Background

- **The three gates are irreversible or human-action-only.** (1) Once a Sparkle-enabled release ships with CFBundleVersion=X, every future release must have a larger integer or Sparkle refuses to update. (2) Once a Sparkle-enabled release ships with SUPublicEDKey=K, losing the corresponding private key means NO FUTURE auto-update will ever reach those installs — full rotation requires 18+ months of dual-key overlap per the rotation procedure in ci-secrets.md. (3) A README notice before v2.2 ships is easier than retrofitting communication after v2.1 users start missing updates.
- **CFBundleVersion = 220 is the specific scheme choice** (Research Pitfall E + A3). Alternatives considered and rejected:
  - `git rev-list --count HEAD` — monotonic by construction but less readable and requires repo access during CI; fine but heavier.
  - `CFBundleShortVersionString.replace('.', '')` (what we chose) — simpler, human-readable, monotonic for any X.Y.Z with each component <10000, documented in sparkle-setup.md.
  - Start-from-1-and-increment — would require state; rejected.
  - Dual-value static `22000` — too many digits, hides the relationship to the semver string; rejected.
- **Sparkle's comparison is lexicographic** (Pitfall #19 in PITFALLS.md). `"2.10"` sorts BEFORE `"2.9"` in ASCII lexicographic order — that's what makes the string scheme a bomb. Integers compared as strings still compare correctly digit-length-first in ASCII: `"220" < "221" < "230" < "1000" < "2100"` all compare correctly as long as the left-most digit reflects magnitude (which strip-dots preserves for any reasonable semver range).
- **Key-generation ritual.** The ritual is once-per-project-lifetime. Steps: (a) developer Mac runs `generate_keys` (interactive, adds private key to login Keychain, prints public key), (b) developer exports private key via `generate_keys -x sparkle_private_key.txt` to a one-line base64 file, (c) paste private key content into GitHub Secrets → `SPARKLE_ED_PRIVATE_KEY`, (d) backup the one-line file to two offline locations (NOT git; NOT 1Password-shared; NOT Slack), (e) update `docs/release/ci-secrets.md` with location descriptions for each backup, (f) delete the local `sparkle_private_key.txt`, (g) commit the public key into `Info.plist`. All of (a)-(g) are operator tasks; Claude Code's role is to present the procedure clearly and confirm each step before moving on.
- **Why README placement matters.** ROADMAP Phase 12 Success Criterion #6 says "README has a one-time manual-update notice for v2.1 users" — PITFALLS.md #9 goes further: without the notice, v2.1 users simply never learn that auto-update exists and stay on v2.1 forever. Place the notice ABOVE the `## Features` section so it's visible above-the-fold; use a callout (`> ...`) to differentiate from marketing copy. Don't make it a scary warning — the user-effect is trivial (download once).
- **The rc.2 dry-run is load-bearing.** Phase 11's CI-13 dry-run used `v2.2.0-rc.1` and exercised the pre-Sparkle pipeline. The rc.2 dry-run is THIS plan's gate — the first time the full pipeline runs (Plan 1 app integration + Plan 2 CI appcast + Plan 3 real key). Failures here reveal integration bugs between the plans that no static check could catch. Capture everything in `12-RC2-DRY-RUN-LOG.md` so future operators have a sample of what "green" looks like.

**Locked decisions (apply verbatim):**
- CFBundleVersion scheme: strip-dots from CFBundleShortVersionString. `2.2.0` → `220`. Pre-release tags strip to the same integer as the final version.
- `set_version.sh` computes the integer from the input. No `rev-list` call. No git dependencies.
- Placeholder replacement in Info.plist (Task 2) uses `PlistBuddy -c "Set :CFBundleVersion 220"` so the repo's checked-in baseline is already migrated before first CI run. CI will stamp over it per-tag anyway, but the baseline must not be a dotted string — otherwise local dev builds fail verify_bundle.sh.
- Key-generation ritual is a single `checkpoint:human-action` task (Task 3). Claude presents the steps; user executes; user pastes public key back into the prompt; Claude resumes to Task 4 (commit the real public key).
- README notice is a Markdown callout, ≤5 lines, placed before `## Features`. Link to `https://github.com/mitgor/mdviewer/releases/latest`.
- rc.2 dry-run is a `checkpoint:human-verify` (Task 7). User pushes tag, watches workflow, manually installs rc.2 DMG, selects "Check for Updates" against a staging appcast, and pastes evidence into the log file. Claude then commits the log.

```bash
#!/usr/bin/env bash
# Usage: set_version.sh <version-string>
# Example: set_version.sh 2.2.0       -> CFBundleShortVersionString=2.2.0,       CFBundleVersion=220
# Example: set_version.sh 2.2.0-rc.2  -> CFBundleShortVersionString=2.2.0-rc.2, CFBundleVersion=220
#
# Stamps CFBundleShortVersionString (human-visible semver) AND CFBundleVersion
# (monotonic integer derived by strip-dots from the short version) into
# MDViewer/Info.plist. The git checkout is mutated in-place; CI never commits
# this change back.
#
# Scheme (SPK-08 migration; locked in Phase 12 Plan 3):
#   - strip any pre-release suffix ("-rc.2", "-beta.1", etc.) from the
#     version -- the integer reflects the FINAL release number, not the
#     intermediate RC number. rc.1 and rc.2 of 2.2.0 both stamp 220.
#     Rationale: rc tags are pre-release and not reachable via /releases/latest/,
#     so the collision is harmless; simpler than a 5-digit scheme.
#   - remove dots from what's left: 2.2.0 -> 220, 2.10.0 -> 2100, 2.2.10 -> 2210.
#     Monotonic for any X.Y.Z with each component <10000. See Research A3.
set -euo pipefail

VERSION="${1:?version required, e.g. 2.2.0 or 2.2.0-rc.2}"
PLIST="MDViewer/Info.plist"

if [ ! -f "$PLIST" ]; then
  echo "::error::$PLIST not found"
  exit 1
fi

# Strip pre-release suffix; then strip dots.
CLEAN_VERSION="${VERSION%%-*}"              # "2.2.0-rc.2" -> "2.2.0"
INTEGER_VERSION="${CLEAN_VERSION//./}"      # "2.2.0" -> "220"

# Sanity-check: result is non-empty and all digits.
if [[ ! "$INTEGER_VERSION" =~ ^[0-9]+$ ]]; then
  echo "::error::Derived integer version is not purely numeric: '$INTEGER_VERSION' (from input '$VERSION')"
  echo "::error::Expected input like '2.2.0' or '2.2.0-rc.2'."
  exit 1
fi

/usr/libexec/PlistBuddy -c "Set :CFBundleShortVersionString $VERSION"      "$PLIST"
/usr/libexec/PlistBuddy -c "Set :CFBundleVersion $INTEGER_VERSION"         "$PLIST"

echo "Stamped $PLIST: ShortVersionString=$VERSION  Version=$INTEGER_VERSION"
/usr/libexec/PlistBuddy -c "Print :CFBundleShortVersionString" "$PLIST"
/usr/libexec/PlistBuddy -c "Print :CFBundleVersion"            "$PLIST"
```

## CFBundleVersion scheme (Phase 12 Plan 03 Task 1-2)

Sparkle compares `CFBundleVersion` **lexicographically** (Pitfall #19). A
dotted-string like `"2.10"` sorts BEFORE `"2.9"` in ASCII, which breaks updates
the moment a minor version exceeds 9. MDViewer's scheme:

- `CFBundleShortVersionString` = human-visible semver: `2.2.0`, `2.2.1`, `2.10.0`, `2.2.0-rc.2`
- `CFBundleVersion` = strip-dots integer: `220`, `221`, `2100`, `220` (pre-release suffix stripped)

Why strip-dots (not git-rev-count): simpler, human-readable, monotonic for any
X.Y.Z with each component <10000. RC collisions accepted (rc.1 and rc.2 both
stamp 220) because RC tags are pre-release and not served from `/releases/latest/`.

The scheme is implemented in `Scripts/set_version.sh` and applied by CI at
build time. Baseline `MDViewer/Info.plist` checked-in value is always an
integer (220 as of 2026-04-20); CI stamps over it per-tag.

**Irreversible:** once v2.2.0 shipped with `CFBundleVersion=220`, every future
release must be a LARGER integer or Sparkle refuses to install.
