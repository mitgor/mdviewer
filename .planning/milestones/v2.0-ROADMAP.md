# Milestone v2.0: Speed & Memory

**Status:** SHIPPED 2026-04-16
**Phases:** 01–05
**Total Plans:** 8 (1 + 2 + 3 + 1 + 1)

## Overview

Memory correctness, true progressive rendering, launch-speed measurement + WKWebView pre-warm, Mermaid script-src loading (eliminating the 3MB IPC bridge call), and per-file window position persistence with proper cascading. The first proper release following v1 — established the OSSignposter instrumentation baseline that v2.1 builds on.

## Phases

### Phase 1: Correctness & Measurement Baseline

**Goal**: Every window close releases all WKWebView memory, and pipeline instrumentation produces valid profiling data.
**Depends on**: Nothing (first phase)
**Requirements**: MEM-01, LAUNCH-01
**Plans**: 1

Plans:
- [x] 01-01: Fix WKWebView retain cycle and add OSSignposter pipeline instrumentation

**Details:**
`WKUserContentController` held a strong reference to `self` via `add(_:name:)`, creating a retain cycle that prevented deallocation on window close. Fixed with `WeakScriptMessageProxy`. Added `OSSignposter` intervals for file-read, parse, chunk-split, and chunk-inject phases. Baseline measurements recorded: cold launch, warm launch, peak RSS for a 10MB file. Verified via deinit logging and Instruments that closing 10 windows in sequence does not grow sustained RSS.

### Phase 2: Large File Memory & Progressive Rendering

**Goal**: Users can open 10MB+ markdown files without memory spikes, and content appears progressively in multiple chunks.
**Depends on**: Phase 1
**Requirements**: MEM-02, RENDER-01, RENDER-02
**Plans**: 2

Plans:
- [x] 02-01: Memory-mapped file read and byte-size N-chunk splitting
- [x] 02-02: Replace evaluateJavaScript with callAsyncJavaScript for typed chunk injection

**Details:**
`String(contentsOf:)` replaced with `Data(contentsOf: fileURL, options: .mappedIfSafe)` — avoids full-file heap allocation. `mappedIfSafe` chosen over `alwaysMapped` for graceful fallback on unsafe volumes. True N-chunk splitting at ≤64KB boundaries (block-level, not byte-level) for visible multi-stage progressive render. Chunk injection migrated to `callAsyncJavaScript(_:arguments:)` with typed arguments — eliminates injection risk from string-interpolated JavaScript.

### Phase 3: Launch Speed

**Goal**: Measure and optimize launch-to-paint pipeline; WKWebView pre-warm implemented and profiled.
**Depends on**: Phase 2
**Requirements**: LAUNCH-02
**Plans**: 3

Plans:
- [x] 03-01: Add launch-to-paint signpost for end-to-end launch measurement
- [x] 03-02: Profile launch path, resolve WKWebView pre-warm decision, record timing data
- [x] 03-03: Gap closure — run Instruments profiling and record actual measured timing data

**Details:**
End-to-end `launch-to-paint` OSSignposter interval added, separately measuring cold vs. warm launch. Pre-warm decision resolved with actual profiling data: ~40ms savings, implemented. Warm launch measured at **184.50ms on M4 Max** — below v2.0's <200ms target but above the <100ms aspiration that v2.1 carries forward.

### Phase 4: Mermaid Script Loading

**Goal**: Mermaid diagrams render without a 3MB IPC bridge call per window.
**Depends on**: Phase 1
**Requirements**: MEM-03
**Plans**: 1

Plans:
- [x] 04-01: Replace evaluateJavaScript Mermaid loading with script-src injection

**Details:**
Previously, the 3MB `mermaid.min.js` was shipped to every window via `evaluateJavaScript`, crossing the WKWebView IPC boundary. Replaced with DOM `createElement('script')` injection so the WebContent process loads the bundle directly from its own resource URL. Windows without Mermaid content no longer load the bundle at all.

### Phase 5: Window Management

**Goal**: Windows remember their position across launches and multiple windows cascade properly.
**Depends on**: Phase 1
**Requirements**: WIN-01, WIN-02
**Plans**: 1

Plans:
- [x] 05-01: Per-file window position persistence and multi-window cascading

**Details:**
`setFrameAutosaveName(_:)` with per-file key format `MDViewer:{absolutePath}` — window position persists across launches keyed to the file. Multiple simultaneous opens use `cascadeTopLeft` with tracking so the second, third, nth window does not overlap the first.

## Milestone Summary

**Key Decisions:**
- Module-level `renderingSignposter` constant (MarkdownRenderer is stateless).
- `mappedIfSafe` over `alwaysMapped` — graceful fallback on unsafe volumes.
- `callAsyncJavaScript` with typed arguments over `evaluateJavaScript` — eliminates string-interpolation injection risk.
- WKWebView pre-warm implemented (~40ms measured savings).
- Script-src injection for `mermaid.min.js` — eliminates 3MB IPC per window.
- Per-file autosave name format `MDViewer:{absolutePath}` — unambiguous per-document persistence.
- `WeakScriptMessageProxy` — breaks WKUserContentController retain cycle without giving up ergonomic delegate pattern.
- `OSSignposter` over raw `os_signpost()` — type-safe interval tracking in Swift.

**Issues Resolved:**
- WKWebView memory not reclaimed on window close (retain cycle via script message handler).
- 10MB+ files spiked heap 2-3× file size (non-mapped String read).
- 3MB `mermaid.min.js` crossed IPC boundary every window load.
- Windows always opened at default position (no persistence).
- Multiple simultaneous opens stacked at same location.

**Issues Deferred:**
- Sub-100ms warm launch target (measured 184.50ms; deferred to v2.1 architectural work).

**Technical Debt Incurred:**
- SPM `swift-cmark` dependency retained (vendoring deferred to v2.1 Phase 6).
- No GitHub Actions release workflow yet (manual archive/sign/export/publish).
