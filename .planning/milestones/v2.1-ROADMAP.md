# Milestone v2.1: Deep Optimization

**Status:** SHIPPED 2026-04-18
**Phases:** 06–09
**Total Plans:** 10 (3 + 2 + 2 + 3)

## Overview

Architectural optimizations beneath the v2.0 memory/streaming foundation: project-owned cmark with a chunked callback API, a warm WKWebView pool for instant subsequent opens, a true streaming parse-to-render pipeline, and a native NSTextView backend that bypasses WebKit for plain files. Shipped as a signed and notarized DMG (https://github.com/mitgor/mdviewer/releases/tag/v2.1).

## Phases

### Phase 6: Vendored cmark

**Goal**: Markdown parsing uses a project-owned cmark with a chunked callback API, eliminating all regex-based HTML post-processing.
**Depends on**: Phase 5 (v2.0 complete)
**Requirements**: CMARK-01..06
**Plans**: 3

Plans:
- [x] 06-01: Vendor cmark-gfm sources and configure static library build targets
- [x] 06-02: Add chunked HTML callback API and mermaid detection to vendored html.c
- [x] 06-03: Rewrite MarkdownRenderer to use chunked C API with cached extensions

**Details:**
cmark-gfm sources vendored under `Vendor/cmark-gfm` and built as two static library targets (`cmark-gfm`, `cmark-gfm-extensions`). Unified modulemap merges both for Xcode explicit-modules compatibility. `html.c` emits ≤64KB chunks at top-level AST boundaries via a C callback; the callback's opaque pointer is `nil` on the non-chunked path, leaving legacy behavior untouched. Mermaid info-string detection and placeholder-div emission moved from Swift regex scan into the C renderer. `MarkdownRenderer` now drives the chunked API directly with a cached extension list, eliminating per-render `cmark_find_syntax_extension` lookups. Entity encoding uses `append(contentsOf:)` with reserved capacity.

### Phase 7: WKWebView Pool

**Goal**: Opening a second (or subsequent) file delivers content to a pre-warmed WKWebView with zero initialization delay.
**Depends on**: Phase 6
**Requirements**: POOL-01..03, PERF-03
**Plans**: 2

Plans:
- [x] 07-01: Create WebViewPool class and add pool support methods to WebContentView
- [x] 07-02: Integrate pool into AppDelegate and add unit tests

**Details:**
`WebViewPool(capacity: 2)` maintains pre-warmed `WebContentView` instances, replenished asynchronously via `DispatchQueue.main.async` after each dequeue. A `WKNavigationDelegate` extension on the pool detects `webViewWebContentProcessDidTerminate` and calls `discard()`/`replenish()` so a crashed WebContent process does not leave the pool empty. First-paint state (`hasProcessedFirstPaint`) resets on view reuse. Signpost lifetime corrected to end the `open-to-paint` interval when `self` is deallocated during chunk injection. Crashed web processes are reported to the user. Fixes: app no longer terminates when the Open panel dismisses (pendingFileOpens guard); UTType matching now accepts `.md` files via imported UTI declarations.

### Phase 8: Streaming Pipeline

**Goal**: First visible content appears while the parser is still producing remaining chunks, closing the gap to sub-150ms warm launch.
**Depends on**: Phase 6
**Requirements**: STRM-01, STRM-02, PERF-01
**Plans**: 2

Plans:
- [x] 08-01: Streaming render API with buffer-reuse template assembly
- [x] 08-02: Wire streaming pipeline into AppDelegate and WebContentView

**Details:**
`renderStreaming` fires `onFirstChunk` mid-C-callback; `AppDelegate` dispatches to the main thread immediately, so WKWebView loads the first page before the AST walk completes. `setRemainingChunks` on `WebContentView` decouples chunk production from injection — the parser can hand over remaining chunks asynchronously. `assembleFirstPage` uses `[UInt8]` with `removeAll(keepingCapacity: true)` to avoid new String allocations on each render. `open-to-paint` OSSignposter interval instruments the path end-to-end.

### Phase 9: Native Text Rendering

**Goal**: Simple markdown files (no mermaid, no GFM tables) render via NSTextView, bypassing WKWebView for dramatically faster display.
**Depends on**: Phase 6
**Requirements**: NATV-01..04, PERF-02
**Plans**: 3

Plans:
- [x] 09-01: NativeRenderer (AST-to-NSAttributedString) and NativeContentView with OTF fonts
- [x] 09-02: Integration: AST pre-scan, AppDelegate routing, MarkdownWindow generalization, toggle menu
- [x] 09-03: NativeRendererTests and visual/performance verification checkpoint

**Details:**
`NativeRenderer` walks the cmark AST via `cmark_iter_new`/`cmark_iter_next` to build an `NSAttributedString` — no HTML intermediate. `NativeContentView` wraps an `NSScrollView` + `NSTextView` with the same Latin Modern Roman OTF fonts (registered once via thread-safe static `let`). An AST pre-scan on `MarkdownRenderer` determines mermaid/table presence so `AppDelegate` can route simple files down the native path. `MarkdownWindow` generalized to host either content view. A View-menu toggle switches native ↔ WebKit per window. Style-stack push/pop balanced for unknown AST node types to prevent attribute bleed. 19 `NativeRendererTests` cover AST-to-attributed-string fidelity.

## Milestone Summary

**Key Decisions:**
- Vendored cmark over SPM swift-cmark (own the C API — chunked callback and mermaid detection require changes Apple's fork couldn't take).
- Unified modulemap merging `cmark_gfm` + `cmark_gfm_extensions` (required by Xcode 26 explicit modules).
- `WebViewPool(capacity: 2)` over single pre-warmed view (covers open-burst scenarios and crash recovery).
- `renderer->opaque` NULL check gates chunked mermaid detection (keeps non-chunked path untouched).
- `Unmanaged.passRetained` for C callback context lifetime safety.
- `pendingFileOpens` guard against premature termination (NSOpenPanel close fires `applicationShouldTerminateAfterLastWindowClosed` before async render completes).
- Zero-copy C-to-JS bridge **deferred** (Phase 8 research showed <0.1ms cost — IPC boundary, not string allocation, is the bottleneck).
- Incremental cmark parsing **ruled out** (`cmark_parser_finish()` is required before AST access; streaming must be post-parse).

**Issues Resolved:**
- WebContent process crashes no longer leave the pool empty.
- `.md` UTI matching now correct for Finder-initiated opens.
- Signpost intervals no longer leak past view deallocation.
- Attribute bleed between siblings in NativeRenderer (style-stack imbalance).
- Force-cast in print handling replaced with `guard let`.

**Issues Deferred:**
- Sub-100ms warm launch target not yet verified on current build — v2.0 measurement was 184.50ms on M4 Max, and v2.1 adds NativeRenderer + streaming + pool which should all help, but no post-v2.1 Instruments measurement recorded.
- Monospace toggle on NativeContentView is a no-op stub (flag-only, no re-render); out of NATV-01..04 scope by design.
- Visual confirmation of Latin Modern Roman rendering on native path (NATV-03) still requires a human check under Instruments.
- Buffer-reuse effectiveness for `assembleFirstPage` under Instruments allocations trace (STRM-02).
- Human sign-off on UAT scenarios for phases 07 (1 pending) and 08 (2 pending).
- Human sign-off on VERIFICATION.md for phases 06, 07, 08, 09 (all `human_needed` at close).

**Technical Debt Incurred:**
- Two rendering backends to maintain (WebKit + Native). Routing decision lives in `AppDelegate` via AST pre-scan.
- No GitHub Actions release workflow — archive/sign/notarize/DMG/release is run manually on the developer Mac.
- v2.0 phase directories (01–05) were partially deleted in the working tree during this milestone but never formally archived to `.planning/milestones/v2.0-ROADMAP.md`. Outstanding cleanup.
