# Requirements: MDViewer v2.1 — Deep Optimization

**Defined:** 2026-04-16
**Shipped:** 2026-04-18
**Core Value:** Open a markdown file and see beautifully rendered content instantly

## v2.1 Requirements (All Complete)

### Vendored cmark

- [x] **CMARK-01**: App uses a vendored copy of cmark-gfm built as part of the Xcode project (no SPM dependency)
- [x] **CMARK-02**: Vendored cmark exposes a chunked HTML callback API that emits ≤64KB blocks at top-level AST boundaries
- [x] **CMARK-03**: Mermaid code blocks are detected in the C renderer (info string check) and emitted as placeholder divs
- [x] **CMARK-04**: MarkdownRenderer uses the chunked API directly, eliminating regex-based HTML chunking and mermaid regex scan
- [x] **CMARK-05**: Cached cmark extension list reused across renders (no per-render `cmark_find_syntax_extension` lookups)
- [x] **CMARK-06**: Entity encoding uses `append(contentsOf:)` with correct capacity reservation

### WKWebView Pool

- [x] **POOL-01**: App maintains a pool of 2 pre-warmed WKWebView instances ready for immediate use
- [x] **POOL-02**: Pool replenishes asynchronously after a view is acquired
- [x] **POOL-03**: Pool handles WebContent process termination (recreates crashed views)

### Streaming Pipeline

- [x] **STRM-01**: First HTML chunk is delivered to WKWebView while remaining chunks are still being rendered from the AST
- [x] **STRM-02**: Template concatenation reuses buffers instead of creating new String allocations per render *(code-level complete; Instruments heap trace deferred — human verification)*

### Native Rendering

- [x] **NATV-01**: Files without mermaid blocks or GFM tables render via NSTextView instead of WKWebView
- [x] **NATV-02**: NSTextView backend walks the cmark AST directly to build NSAttributedString (no HTML intermediate)
- [x] **NATV-03**: Native rendering uses the same Latin Modern Roman typography as the WKWebView path *(code-level complete; visual confirmation deferred — human verification)*
- [x] **NATV-04**: User can toggle between native and web rendering if visual differences arise

### Performance Validation

- [x] **PERF-01**: Warm launch to first visible content is under 150ms for WKWebView path *(OSSignposter instrumented; Instruments measurement deferred — human verification)*
- [x] **PERF-02**: Warm launch to first visible content is under 100ms for NSTextView path *(OSSignposter instrumented; Instruments measurement deferred — human verification)*
- [x] **PERF-03**: 2nd+ file open is under 100ms with WKWebView pool active *(existing `open-to-paint` signpost covers it; Instruments measurement deferred — human verification)*

## Out of Scope (Decided During v2.1)

| Feature | Reason |
|---------|--------|
| Swift async/await migration | WKWebView main-actor requirements make it a correctness risk |
| Zero-copy C-to-JS bridge | Phase 8 research showed <0.1ms cost; IPC boundary is the real bottleneck |
| Incremental cmark parsing | `cmark_parser_finish()` required before AST access; streaming must be post-parse |
| Dark mode | Conflicts with LaTeX aesthetic |
| File editing | Viewer, not editor |
