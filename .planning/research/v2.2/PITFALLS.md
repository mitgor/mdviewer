# PITFALLS: MDViewer v2.2 Release Quality & Automation

**Domain:** macOS app release automation — GitHub Actions notarization, Sparkle 2.x auto-update, Homebrew cask publishing, Instruments measurement
**Researched:** 2026-04-18
**Project context:** MDViewer is unsandboxed (hardened runtime only), Developer ID Application "Woodenshark LLC (V7YK72YLFF)", two certs share that CN, vendored cmark-gfm static lib, WKWebView, macOS 13+
**Overall confidence:** MEDIUM-HIGH (most claims verified against official Sparkle/Apple docs and reproducible GitHub issues; speculative items flagged)

## CRITICAL PITFALLS — Will Block Release Pipeline

### 1. Ambiguous Developer ID Application Certificates with Identical CN

**What goes wrong:** `codesign --sign "Developer ID Application: Woodenshark LLC (V7YK72YLFF)"` fails with "ambiguous (matches more than one identity)" when two certs share the CN. This is MDViewer's exact situation per project context.

**Why:** When the identity argument matches more than one item in the keychain, codesign refuses to guess. macOS does not deduplicate certificates by CN.

**Mitigation:** Use the SHA-1 (40 hex digits) of the desired cert as the identity. From `codesign(1)`: "If [identity] consists of exactly forty hexadecimal digits, it is instead interpreted as the SHA-1 hash of the certificate part of the desired identity." Workflow:
```bash
# Fetch SHA-1 of valid cert (filter by team ID + not expired)
SIGNING_ID=$(security find-identity -v -p codesigning build.keychain \
  | grep "V7YK72YLFF" | head -n1 | awk '{print $2}')
codesign --sign "$SIGNING_ID" --options runtime --timestamp ...
```
Store `SIGNING_ID` early in the job; use the same value for every signing step (XPC services, framework, app bundle, DMG). Never use the CN string in CI.

**Citation:**
- [codesign(1) man page](https://keith.github.io/xcode-man-pages/codesign.1.html)
- [Apple Developer Forum: Ambiguous identity discussion](https://developer.apple.com/forums/thread/750000)
- [Stack Overflow / SAP Community: SDK 5.0 Handling Ambiguous Identities](https://blogs.sap.com/2020/03/09/sdk-5.0-handling-ambiguous-apple-developer-identity)

### 2. EdDSA Private Key Loss = Permanent Lock-Out for Existing Installs

**What goes wrong:** If you lose the EdDSA private key generated by `generate_keys`, you cannot sign updates that existing installs will accept. Users running the old version will get "Update error. Failed to extract update" indefinitely. Issue [#1695](https://github.com/sparkle-project/Sparkle/issues/1695) shows exactly this scenario with no maintainer-provided recovery procedure in the issue thread.

**Why:** Sparkle verifies the appcast's `sparkle:edSignature` against the `SUPublicEDKey` baked into the *currently installed* app's Info.plist. If the new private key doesn't match the embedded public key, validation fails and Sparkle refuses to install. There is no "recovery channel" — the public key is immutable for that install.

**Mitigation (preventive — there is no cure):**
1. After running `generate_keys`, **immediately** export the private key with `generate_keys -x ed_priv.key`, store in 1Password / a hardware token, and back it up to two physical locations.
2. The key is stored in the macOS login keychain — back up the keychain to encrypted external storage.
3. For CI: do NOT use the keychain copy. Export to a base64 secret in GitHub Secrets so CI can sign without keychain access (avoids the user-prompt blocker described in pitfall #14).
4. Document the key fingerprint in a sealed-envelope physical document.
5. **Failsafe:** Sparkle 2 will eventually fall back to presenting unsigned updates to users *if signing has failed for a long enough period* — this is the "lose access to your EdDSA signing keys" failsafe, but it requires user trust on first launch and is not documented as immediate.

**Citation:**
- [Sparkle issue #1695: Lost private key](https://github.com/sparkle-project/Sparkle/issues/1695)
- [Sparkle Documentation: Publishing](https://sparkle-project.org/documentation/publishing/)
- [Sparkle issue #1535: Generate-Keys Tool: Add "Export Key" capability](https://github.com/sparkle-project/Sparkle/issues/1535)
- [VibeTunnel Sparkle keys docs](https://docs.vibetunnel.sh/mac/docs/sparkle-keys)

### 3. `--deep` Flag Corrupts Sparkle XPC Service Signatures

**What goes wrong:** Adding `--deep` to `OTHER_CODE_SIGN_FLAGS`, or running `codesign --deep --force --sign ID MDViewer.app`, will corrupt Sparkle's nested XPC service signatures and produce "Failed to gain authorization required to update target" at runtime. The notarization may even succeed; the app will fail to update silently.

**Why:** Sparkle's Downloader.xpc is signed with specific entitlements (e.g., `com.apple.security.network.client`) that the app bundle does not have. `--deep` overwrites the XPC service signature with the parent app's identity and entitlements, breaking the explicit per-component signing model.

**Mitigation:** Sign each component manually in this exact order (per Sparkle docs and Steipete's "Sparkle and Tears" post):
1. `Sparkle.framework/.../Updater.app/Contents/MacOS/Updater`
2. `Sparkle.framework/.../Autoupdate`
3. `Sparkle.framework/.../XPCServices/Installer.xpc`
4. `Sparkle.framework/.../XPCServices/Downloader.xpc` — **with `--preserve-metadata=entitlements`** (Sparkle 2.6+)
5. `Sparkle.framework` itself
6. App bundle (no `--deep`)
7. DMG

For MDViewer, since the app is unsandboxed and likely won't enable Downloader.xpc (network entitlement isn't a constraint), the order is still required but `--preserve-metadata=entitlements` is less load-bearing.

**Citation:**
- [Sparkle Documentation: Sandboxing](https://sparkle-project.org/documentation/sandboxing/)
- [Steipete: Code Signing and Notarization — Sparkle and Tears (2025)](https://steipete.me/posts/2025/code-signing-and-notarization-sparkle-and-tears)
- [Christian Tietze: Xcode Project Setup for Sparkle XPC Services](https://christiantietze.de/posts/2019/06/sparkle-xpc-setup/)

### 4. `notarytool` "Invalid" Status with Empty `statusMessage`

**What goes wrong:** `xcrun notarytool submit ... --wait` returns "Current status: Invalid" but the top-level `statusMessage` is empty. CI logs show no error context. This is described in [fastlane #20578](https://github.com/fastlane/fastlane/issues/20578) as "very unhelpful error message 'Could not notarize package with message ''".

**Why:** `notarytool submit` only returns top-level status. Detailed errors (signature problems, missing hardened runtime on a nested binary, expired Developer ID, etc.) live in a separate JSON log fetched by submission ID.

**Mitigation:** Always run `xcrun notarytool log <submission-id> --keychain-profile <profile>` after a non-Accepted status. In CI:
```bash
SUBMISSION_OUTPUT=$(xcrun notarytool submit MDViewer.dmg --apple-id ... --team-id V7YK72YLFF --password ... --wait --output-format json)
SUBMISSION_ID=$(echo "$SUBMISSION_OUTPUT" | jq -r '.id')
STATUS=$(echo "$SUBMISSION_OUTPUT" | jq -r '.status')
if [ "$STATUS" != "Accepted" ]; then
  xcrun notarytool log "$SUBMISSION_ID" --apple-id ... --team-id V7YK72YLFF --password ...
  exit 1
fi
```
Common log payload errors:
- "The binary is not signed with a valid Developer ID certificate"
- "The signature does not include a secure timestamp" (missing `--timestamp`)
- "The executable does not have the hardened runtime enabled" (missing `--options runtime`)

**Citation:**
- [fastlane issue #20578: Notarize action can fail without error message](https://github.com/fastlane/fastlane/issues/20578)
- [electron-builder issue #8258: Notarize JSON parsing error](https://github.com/electron-userland/electron-builder/issues/8258)
- [notarytool(1) man page](https://keith.github.io/xcode-man-pages/notarytool.1.html)
- [Apple: Resolving common notarization issues](https://developer.apple.com/documentation/security/resolving-common-notarization-issues)

### 5. `security import` Without `-T /usr/bin/codesign` → ACL Blocks Signing

**What goes wrong:** The imported `.p12` is in the keychain, `security find-identity -v -p codesigning` lists it, but `codesign` fails with "User interaction is not allowed" or hangs waiting for a GUI password prompt that never appears in headless CI.

**Why:** Since macOS 10.12.5, keychain items have a "partition list" ACL. Imported certs default to "no ACL → prompt user". In CI there's no user → indefinite hang or denied access. Two flags interact:
- `-T /usr/bin/codesign` — adds codesign to the trusted apps list at import.
- `-A` — allow ANY app to access (insecure, removed from `Apple-Actions/import-codesign-certs` v5.0.0).

**Mitigation:** After `security import`, run `security set-key-partition-list -S apple-tool:,apple:,codesign: -s -k "$KEYCHAIN_PWD" build.keychain`. This is required in addition to `-T`. Standard recipe:
```bash
security create-keychain -p "$KEYCHAIN_PWD" build.keychain
security set-keychain-settings -lut 21600 build.keychain
security unlock-keychain -p "$KEYCHAIN_PWD" build.keychain
security list-keychains -d user -s build.keychain $(security list-keychains -d user | sed s/\"//g)
security import cert.p12 -k build.keychain -P "$P12_PWD" -T /usr/bin/codesign -T /usr/bin/security
security set-key-partition-list -S apple-tool:,apple:,codesign: -s -k "$KEYCHAIN_PWD" build.keychain
```

**Citation:**
- [Apple-Actions/import-codesign-certs issue #60: Options to security import are confusing](https://github.com/apple-actions/import-codesign-certs/issues/60)
- [Apple Developer Forum: Why is security set-key-partition-list needed?](https://developer.apple.com/forums/thread/666107)
- [Localazy: How to automatically sign macOS apps using GitHub Actions](https://localazy.com/blog/how-to-automatically-sign-macos-apps-using-github-actions)
- [Apple-Actions/import-codesign-certs releases (v5.0.0 dropped -A)](https://github.com/Apple-Actions/import-codesign-certs/releases)

### 6. Sparkle Refuses HTTP `SUFeedURL` Even With NSAllowsArbitraryLoads

**What goes wrong:** Setting `SUFeedURL` to `http://...` and adding `NSAppTransportSecurity` exception → silent failure. Sparkle treats HTTP feed URLs as a security violation regardless of ATS configuration.

**Why:** Sparkle 2.x bypasses the app's ATS settings for the feed URL specifically. EdDSA over HTTP would still be safe cryptographically, but Sparkle imposes HTTPS as a defense-in-depth requirement to prevent MITM tampering with the appcast itself (e.g., suppressing security updates by lying about latest version).

**Mitigation:** Always serve appcast over HTTPS. For MDViewer, the appcast lives at `https://github.com/mitgor/mdviewer/...` or `https://mitgor.github.io/mdviewer/appcast.xml` — both naturally HTTPS via GitHub Pages. **Also verify that every `<enclosure url=...>` inside the appcast is HTTPS** — `generate_appcast` will copy whatever URL you supply, and a single HTTP enclosure will fail download.

**Citation:**
- [Sparkle issue #1856: Refuses http SUFeedURL with NSAllowsArbitraryLoads](https://github.com/sparkle-project/Sparkle/issues/1856)
- [Sparkle Documentation: App Transport Security](https://sparkle-project.org/documentation/app-transport-security/)
- [Sparkle issue #1461: appcast enclosure url is using http but my SUFeedURL is https](https://github.com/sparkle-project/Sparkle/issues/1461)

### 7. Stapler Error 65: cdhash Mismatch Between Submitted and Stapled Artifact

**What goes wrong:** `xcrun stapler staple MDViewer.dmg` fails with "Error 65: ... could not find ticket". Even though notarization was Accepted, stapling refuses.

**Why:** Apple's notary service indexes tickets by cdhash. If you re-sign, repackage, or convert the DMG between submission and stapling, the cdhash changes and there's no ticket for the new hash. Common causes:
- Re-signing the DMG after notarization
- `hdiutil convert` to a different UDIF variant (UDZO ↔ UDRO)
- `hdiutil udifrez` to attach an SLA after signing
- Modifying *any* byte of the artifact

**Mitigation:** Notarize the *exact* artifact you ship. For MDViewer DMG flow:
1. Build app → sign all components in order (per pitfall #3)
2. `hdiutil create -srcfolder ... -format UDZO MDViewer.dmg`
3. `codesign --sign $SIGNING_ID --timestamp MDViewer.dmg`
4. `xcrun notarytool submit MDViewer.dmg --wait`
5. **Without modification:** `xcrun stapler staple MDViewer.dmg`
6. Verify: `spctl -a -t open --context context:primary-signature -vv MDViewer.dmg`

If error 65 occurs, compare `shasum -a 256` of the actual DMG to the `sha256` field in `xcrun notarytool log` output — they must match exactly.

**Citation:**
- [Apple Developer Forum: Notarization stapler error 65](https://developer.apple.com/forums/thread/115670)
- [Apple Developer Forum: Stapler issue Error 65](https://developer.apple.com/forums/thread/123806)
- [electron/notarize issue #109: Failed to staple your application with code 64](https://github.com/electron/notarize/issues/109)

### 8. Apple Notary Service Outages — No SLA, Hours-Long Stalls

**What goes wrong:** `notarytool submit --wait` hangs for hours. Status sticks at "In Progress" with `notarytool log` returning "not available". Recent (Feb–April 2026) reports show 5+ day stalls blocking release.

**Why:** Apple's notary service has no published SLA. Service-level issues cause queue backups. New developer accounts and large submissions are deprioritized. Status page may show green while submissions are stuck.

**Mitigation:**
- **Don't block the GitHub Actions job indefinitely.** Use `--wait` with a reasonable timeout (e.g., wrapping with `timeout 30m`) and design retry logic.
- Check `https://developer.apple.com/system-status/` before declaring failure.
- Submit and detach: omit `--wait`, store the submission ID, and have a separate workflow poll `notarytool info` and run stapling when complete.
- For MDViewer's expected submission size (small, ~10MB DMG), normal turnaround is 1–5 minutes; treat anything past 30 minutes as a service issue, not a problem with the artifact.

**Citation:**
- [electron/notarize issue #179: Notarization hangs on notarytool submit for hours](https://github.com/electron/notarize/issues/179)

### 9. Migration Pitfall: Existing v2.1 Installs Don't Know About Sparkle

**What goes wrong:** Today's MDViewer v2.1 has no Sparkle integration. Users with v2.1 installed will not see v2.2 updates because their app has no `SUFeedURL` and no Sparkle framework. They are permanently stranded on v2.1 unless they manually re-download.

**Why:** Sparkle is opt-in per-version. The installed version of the app is what decides whether to check for updates — the appcast can't reach back to non-Sparkle installs.

**Mitigation:** Three-step rollout:
1. **v2.2 release** = "First Sparkle release". Ship via the existing manual channel (GitHub releases). Existing v2.1 users must download manually one last time.
2. Announce in the v2.2 release notes: "After installing v2.2, future updates will be automatic."
3. From v2.3 onward, Sparkle handles updates seamlessly.
4. For Homebrew users, the cask transition is automatic since `brew upgrade` doesn't depend on Sparkle.
5. Consider keeping manual download links available for at least one major version cycle.

**Note:** `sparkle:minimumUpdateVersion` (Sparkle 2.9+) doesn't help here — it's for filtering forward, not reaching backward.

**Citation:**
- [Sparkle Documentation: Upgrading from previous versions](https://sparkle-project.org/documentation/upgrading/)
- [Sparkle Documentation: Publishing](https://sparkle-project.org/documentation/publishing/)

## MODERATE PITFALLS — Will Slow Pipeline / Break Edge Cases

### 10. Homebrew Cask Without `auto_updates true` → Brew Fights Sparkle

**What goes wrong:** Cask shows "outdated" every time Sparkle has updated the app, because Homebrew sees `app.version` (e.g., 2.3.0 from Sparkle's silent update) ≠ `cask.version` (still 2.2.0 until next bump-cask-pr). `brew upgrade` re-installs the older bundled DMG version, downgrading the user.

**Why:** Without `auto_updates true`, livecheck assumes Homebrew is the canonical source of truth. The presence of Sparkle changes that — the app updates itself between cask bumps.

**Mitigation:** Always declare `auto_updates true` in the cask file when shipping with Sparkle:
```ruby
cask "mdviewer" do
  version "2.2.0"
  sha256 "abc123..."
  url "https://github.com/mitgor/mdviewer/releases/download/v#{version}/MDViewer-#{version}.dmg"
  name "MDViewer"
  desc "Fast native macOS markdown viewer"
  homepage "https://github.com/mitgor/mdviewer"

  auto_updates true
  livecheck do
    url "https://mitgor.github.io/mdviewer/appcast.xml"
    strategy :sparkle
  end

  app "MDViewer.app"
  zap trash: ["~/Library/Preferences/com.mdviewer.app.plist"]
end
```
Homebrew has an active cleanup tracker for this exact mismatch — your cask will be flagged in [issue #170994](https://github.com/Homebrew/homebrew-cask/issues/170994) if you forget.

**Citation:**
- [Homebrew/homebrew-cask issue #170994: Casks with Sparkle/Electron livecheck and no auto_updates true](https://github.com/Homebrew/homebrew-cask/issues/170994)
- [Homebrew Cask Cookbook](https://docs.brew.sh/Cask-Cookbook)

### 11. `brew bump-cask-pr` Race Condition With GitHub Release Publishing

**What goes wrong:** GitHub Actions workflow flow: (1) build & sign DMG, (2) `gh release create`, (3) immediately `brew bump-cask-pr` in next step. The bump tries to download the DMG to compute SHA256 and gets a 404 — the release asset isn't fully propagated through GitHub's CDN.

**Why:** GitHub's release asset upload is eventually consistent. The asset URL becomes valid within seconds of upload, but CDN propagation can lag. `brew bump-cask-pr` calls the URL directly; a stale CDN edge returns 404.

**Mitigation:**
- Insert a `sleep 30` or polling loop between `gh release create` and the bump step.
- Better: poll the URL until `curl -I -L` returns 200.
- Best: pre-compute SHA256 in the build job and pass it explicitly via `brew bump-cask-pr --sha256=<hash>` to skip the download check entirely.
```bash
ASSET_SHA=$(shasum -a 256 MDViewer.dmg | awk '{print $1}')
brew bump-cask-pr --version="$VERSION" --sha256="$ASSET_SHA" mitgor/tap/mdviewer
```

**Citation:**
- [Homebrew/brew issue #15342: brew bump-formula-pr and bump-cask-pr fail](https://github.com/Homebrew/brew/issues/15342)
- [Homebrew/brew issue #11485: Issue with brew bump-cask-pr](https://github.com/Homebrew/brew/issues/11485)
- [Homebrew/brew issue #93013: bump-cask-pr does not work for SHA-only changes](https://github.com/Homebrew/homebrew-cask/issues/93013)

### 12. `brew bump-cask-pr` PAT Scope Requirements

**What goes wrong:** Workflow uses default `GITHUB_TOKEN` → `brew bump-cask-pr` fails because it needs to fork the cask tap repo and create a PR across orgs. Or uses a PAT with `repo` scope but Homebrew needs `workflow` too because the cask repo may have workflow files.

**Why:** `bump-cask-pr` operations span: (a) forking, (b) pushing to fork, (c) creating PR to upstream. Default `GITHUB_TOKEN` is scoped to the workflow's own repo only.

**Mitigation:** Create a fine-grained PAT with at minimum `public_repo` and `workflow` scopes (and `gist` if Homebrew docs are followed strictly). Store as `HOMEBREW_GITHUB_API_TOKEN` secret. For MDViewer's case shipping to `mitgor/homebrew-tap` (your own tap, not Homebrew core), the scope is the same but the destination is your own org — still need a PAT, since `GITHUB_TOKEN` from the source repo can't push to a different repo.

For the tap-only case (you control both repos), an alternative: use a deploy key on `homebrew-tap` and have CI directly push the cask file change rather than using `bump-cask-pr`. Simpler, fewer moving parts.

**Citation:**
- [Homebrew Discussion #5129: Run brew bump-formula-pr with GitHub App Token](https://github.com/orgs/Homebrew/discussions/5129)
- [Homebrew Discussion #4389: brew bump-formula-pr with fine-grained tokens](https://github.com/orgs/Homebrew/discussions/4389)
- [Homebrew/brew issue #17000: Support GitHub App installation access tokens](https://github.com/Homebrew/brew/issues/17000)

### 13. App Store Connect API Key (.p8) Newline Mangling in CI Secrets

**What goes wrong:** Pasting the raw `.p8` content into a GitHub Secret strips or normalizes newlines depending on how it was copied. `notarytool` then fails authentication with "invalid private key format".

**Why:** PEM-formatted keys (BEGIN/END markers around base64) require LF newlines. GitHub Secrets preserves the literal content but only if paste preserves them — copying from `cat` in some terminals or from the App Store Connect web UI may strip newlines.

**Mitigation:** Always base64-encode the key before storing as a secret:
```bash
# Local one-time setup:
base64 -i AuthKey_ABC123.p8 | pbcopy
# Paste into GitHub Secret APP_STORE_CONNECT_API_KEY_BASE64

# In workflow:
echo "${{ secrets.APP_STORE_CONNECT_API_KEY_BASE64 }}" | base64 --decode > AuthKey.p8
xcrun notarytool submit MDViewer.dmg \
  --key AuthKey.p8 \
  --key-id "${{ secrets.APP_STORE_CONNECT_KEY_ID }}" \
  --issuer "${{ secrets.APP_STORE_CONNECT_ISSUER_ID }}" \
  --wait
```
Three secrets total: base64 key, key ID, issuer ID. Issuer ID is a UUID (one per team); key ID is 10 chars (one per key) — easy to swap by accident.

**Required role:** Use a **Team API Key** (not Individual) with at minimum **Developer** role. Individual/Personal API Keys are not eligible for the Notary API.

**Citation:**
- [fastlane Discussion #17930: Running into issues with base64 encoding my App Store Connect API key](https://github.com/fastlane/fastlane/discussions/17930)
- [fastlane app_store_connect_api_key docs](https://docs.fastlane.tools/actions/app_store_connect_api_key/)
- [Apple Developer Forum #133063: Which API Key Role to Use for Notarization](https://developer.apple.com/forums/thread/133063)
- [App Store Connect API Key Roles & Permissions](https://aso.dev/app-store-connect/api-key-access-levels/)

### 14. `generate_appcast` Requires Keychain User Prompt (CI Blocker)

**What goes wrong:** `generate_appcast` reads the EdDSA private key from the macOS login keychain and triggers a GUI password prompt. In CI, no GUI → indefinite hang or "User interaction is not allowed" error.

**Why:** `generate_appcast`'s default key-storage model assumes a developer's interactive Mac. In CI the keychain doesn't even have the key stored; it has to be loaded fresh from a secret.

**Mitigation:** Use the explicit private-key-file flag (CLI `-s` is deprecated; use stdin or `--ed-key-file` per recent Sparkle changes):
```bash
# Decode private key from secret
echo "${{ secrets.SPARKLE_ED_PRIVATE_KEY_BASE64 }}" | base64 --decode > /tmp/sparkle_ed_priv.key
chmod 600 /tmp/sparkle_ed_priv.key

# Sign without keychain
./bin/sign_update --ed-key-file /tmp/sparkle_ed_priv.key MDViewer-2.2.0.dmg
# Or with generate_appcast for a folder of releases
./bin/generate_appcast --ed-key-file /tmp/sparkle_ed_priv.key ./releases/
```
Make sure to delete the key file at the end of the job. Also note: process substitution (`<(echo ...)`) had a known bug in older Sparkle versions — see [PR #2615](https://github.com/sparkle-project/Sparkle/pull/2615); use a real file or stdin.

**Citation:**
- [Sparkle issue #1432: Generate Appcast with Travis-CI](https://github.com/sparkle-project/Sparkle/issues/1432)
- [Sparkle issue #1473: sign_update have problem in CI environment](https://github.com/sparkle-project/Sparkle/issues/1473)
- [Sparkle issue #1364: How to Store private key for signing sparkle updates](https://github.com/sparkle-project/Sparkle/issues/1364)

### 15. Sparkle XPC Service Team ID Hardening

**What goes wrong:** Sparkle 2 hardened policy requires the connecting client (your app) to be code-signed with the same Team ID as the XPC services. Mixing certs from different teams (e.g., dev cert for app, distribution cert for framework) → XPC connection rejected at runtime, "Failed to gain authorization required to update target".

**Why:** Per Sparkle 2 hardening (PR [#2763](https://github.com/sparkle-project/Sparkle/pull/2763)), `xpc_connection_set_peer_team_identity_requirement` is enforced. Different team IDs across the app/framework/XPC trio breaks the trust chain.

**Mitigation:** Sign every component (app, Sparkle.framework, Updater.app, both XPCs, Autoupdate, DMG) with the **same** Developer ID Application certificate. For MDViewer with team `V7YK72YLFF`, the same SHA-1 from pitfall #1 is used everywhere. Validate post-sign:
```bash
codesign -dvvv MDViewer.app | grep "TeamIdentifier"
codesign -dvvv MDViewer.app/Contents/Frameworks/Sparkle.framework | grep "TeamIdentifier"
# Both should show: TeamIdentifier=V7YK72YLFF
```

**Citation:**
- [Sparkle PR #2763: Harden policy on what operations clients are allowed to take](https://github.com/sparkle-project/Sparkle/pull/2763)
- [Sparkle Documentation: Sandboxing](https://sparkle-project.org/documentation/sandboxing/)
- [Apple: xpc_connection_set_peer_team_identity_requirement](https://developer.apple.com/documentation/xpc/xpc_connection_set_peer_team_identity_requirement(_:_:))

### 16. xctrace Cannot Enable Signposts From CLI Headlessly

**What goes wrong:** Plan was to run Instruments in CI to capture `OSSignposter(open-to-paint)` measurements for PERF-01..03 verification. `xctrace record` from a GitHub Actions macOS runner produces a `.trace` file but signpost data is missing — only the dynamic-tracing-enabled instruments (Time Profiler, Allocations) work.

**Why:** Per [explainx.ai/skills/charleswiltgen/axiom](https://explainx.ai/skills/charleswiltgen/axiom/axiom-xctrace-ref) and Apple Developer Forum threads, **xctrace does not provide an option to enable dynamic tracing (signposts) from within the command-line.** This must be configured through the Instruments GUI's recording options. CI runners are headless → can't reach the GUI configuration.

**Mitigation:** Two viable options:
1. **Don't measure perf in CI.** Reserve Instruments runs for the developer Mac. Document a recurring "perf measurement" checklist in the milestone process; verify on M4 Max locally.
2. **Use a custom `.tracetemplate`** built once via the GUI on a developer Mac, committed to the repo, then invoked in CI:
   ```bash
   xcrun xctrace record --template ./perf/MDViewer-Signposts.tracetemplate \
     --launch -- /path/to/MDViewer.app/Contents/MacOS/MDViewer \
     --output ./trace-output.trace
   xcrun xctrace export --input trace-output.trace --xpath '//table[@schema="os-signpost"]' > signposts.xml
   ```
   The `.tracetemplate` carries the signpost configuration past the CLI's limitation.
3. **Even with templates, runner architecture matters.** GitHub Actions macOS runners may not match the perf profile of an M4 Max — performance numbers from CI are not authoritative for SLO checks. Treat CI perf as a regression gate (did we get *much worse*?) not as an absolute measurement.

### 17. macOS Runner Cost: 10× Multiplier vs Linux

**What goes wrong:** Burning through GitHub Actions free minutes faster than expected. Standard Intel macOS runner is **$0.08/min** = **10× the Linux multiplier** (1 macOS minute consumes 10 of your included 2000 monthly minutes on free plan). M1 6-core larger runner is $0.16/min = 20× multiplier.

**Why:** macOS runners require dedicated VMs on Apple hardware — Apple licensing forbids virtualization on non-Apple hosts. GitHub passes through the cost.

**Mitigation:**
- Free tier gets ~300 actual macOS minutes/month (3000 minutes after multiplier on the older billing model). At 20 min/release × 2 releases/month = 40 min/month — well within free tier.
- Cache DerivedData to halve build times (see pitfall #18).
- Don't run the full notarization pipeline on every PR; gate it on tag pushes only.
- January 2026 GitHub announced a price reduction of "up to 39%" effective — verify current pricing at billing time.
- M1 (arm64) runners are 2× more expensive per minute but ~2-4× faster for Swift compilation; net cost may be similar or favorable.

### 18. DerivedData Cache Limited Effectiveness on Hosted Runners

**What goes wrong:** Caching `~/Library/Developer/Xcode/DerivedData` between runs to speed up incremental builds. Each run still triggers a full rebuild. Cache hit rate is high but build time barely improves.

**Why:** Xcode tracks file modification time at **nanosecond resolution** to decide what to recompile. `actions/cache` doesn't preserve nanosecond mtimes — files restored from cache get fresh mtimes, so Xcode invalidates everything anyway. Standard caching is essentially useless for Xcode's incremental engine.

**Mitigation:** Use [`irgaly/xcode-cache`](https://github.com/irgaly/xcode-cache) which captures and restores mtimes via a sidecar manifest. Reported to reduce build times by 40-50%.

For MDViewer specifically:
- The full clean build is small (single Swift target + vendored cmark-gfm C lib). Probably under 60s on M1 runner.
- Caching DerivedData may save 30s but adds complexity.
- Verify whether it's worth it before adding.
- DerivedData can be huge (multi-GB); cache size counts against the 10GB per-repo cache budget.

### 19. Sparkle Generates Wrong Build Number → Updates Compared as Strings

**What goes wrong:** `generate_appcast` writes `sparkle:version="1.0"` because it can't extract the real `CFBundleVersion`. Sparkle then compares versions lexicographically — "10" < "2" — so users see "no update available" or get downgraded.

**Why:** `CFBundleVersion` should be monotonically increasing build numbers (e.g., "1248", "1249"). `CFBundleShortVersionString` is the human-readable "2.2.0". Sparkle compares CFBundleVersion. If you're using string-style version numbers in CFBundleVersion, you're playing with fire.

**Mitigation:**
- Use **integer build numbers** in `CFBundleVersion` (e.g., monotonic counter, or auto-increment from CI tag count).
- Use **human-readable** versions in `CFBundleShortVersionString` ("2.2.0").
- Custom version comparators were deprecated in **Sparkle 2.7** — don't rely on them.
- For three-part `minimumSystemVersion` ("13.0.0" not "13.0") — Sparkle requires major.minor.patch.
- After `generate_appcast`, manually inspect the XML to verify `sparkle:version` reflects the true build number.

For MDViewer: existing `Info.plist` has `CFBundleVersion = 1.0` and `CFBundleShortVersionString = 1.0` — both are "1.0". This needs to change before Sparkle ships. Move CFBundleVersion to an integer scheme (say "10000" for v1.0, "20000" for v2.0, "22000" for v2.2) early so future updates compare correctly.

**Citation:**
- [Sparkle issue #1315: generate_appcast just generates version 1.0](https://github.com/sparkle-project/Sparkle/issues/1315)
- [Sparkle Documentation: Publishing](https://sparkle-project.org/documentation/publishing/)
- [Sparkle CHANGELOG (2.x): Custom version comparators deprecated](https://github.com/sparkle-project/Sparkle/blob/2.x/CHANGELOG)
- [WinSparkle Appcast Feeds wiki](https://github.com/vslavik/winsparkle/wiki/Appcast-Feeds)

### 20. Stapling vs DMG Format Choice

**What goes wrong:** Notarized a flat `.app` zip submission, then tried to package the app into a DMG and ship that — the DMG itself is unsigned/unstapled even though the inner app has the ticket. Gatekeeper warns user.

**Why:** Stapling attaches the ticket to the artifact; the inner app's ticket isn't visible to Gatekeeper checking the DMG. Notary service accepts DMGs (UDIF) directly — submit the DMG, get a ticket on the DMG.

**Mitigation:** Submit the DMG directly to notarytool. The order:
1. Sign the app bundle (with `--options runtime --timestamp`)
2. `hdiutil create ... -format UDZO MDViewer.dmg`
3. `codesign --sign $SIGNING_ID --timestamp MDViewer.dmg` (DMG itself is signed)
4. `xcrun notarytool submit MDViewer.dmg --wait`
5. `xcrun stapler staple MDViewer.dmg`

The app bundle inside doesn't strictly need its own staple if the DMG is stapled — the user mounts the DMG, drags the app, Gatekeeper checks the app, finds it signed by Developer ID, and the staple on the DMG provides offline notarization confirmation (ticket cached after first launch).

For maximum offline robustness, consider stapling **both**: staple the app bundle pre-DMG-creation, then submit/staple the DMG. This means two notary submissions (one for app zip, one for DMG) — twice the notary wait time but bulletproof.

**Citation:**
- [LiveCode Lessons: Code signing and notarizing your standalone with Xcode 13+](https://lessons.livecode.com/a/1653720-code-signing-and-notarizing-your-lc-standalone-for-distribution-outside-the-mac-appstore-with-xcode-13-and-up)
- [OOZOU: Scripting Notarization for macOS App Distribution](https://oozou.com/blog/scripting-notarization-for-macos-app-distribution-38)
- [Apple Developer Forum #134060: Notarize dmg failed](https://developer.apple.com/forums/thread/134060)

## MINOR PITFALLS — Worth Knowing, Lower Impact

### 21. Vendored cmark-gfm Static Lib & Sparkle Delta Updates

**What goes wrong:** Concern: vendored `cmark-gfm` static archive embedded in MDViewer binary creates a large binary blob. Will Sparkle's binary delta updates regenerate huge patches when only Swift code changes?

**Why (LOW confidence — based on inference):** Static library archives are linked into the final Mach-O. Once linked, the lib's bytes are interleaved with Swift code and depend on linker output ordering. Small Swift changes can shift offsets in the final binary, making the binary diff appear larger than the source change.

**Mitigation:**
- Sparkle's BinaryDelta v4 (default since [PR #2668](https://github.com/sparkle-project/Sparkle/pull/2668)) uses bsdiff which handles offset shifts better than naive diff.
- For MDViewer's expected binary size (probably under 5MB), even a "bad" delta is small compared to the full DMG.
- Sparkle's docs explicitly say: "Content that you expect to change in an update should be stored in separate files from content that you don't expect to change" — but for a single Mach-O this isn't actionable.
- **Don't pre-optimize.** Ship full-update appcasts initially. Add delta updates in a future milestone if the full DMG becomes a meaningful download size (>50MB).

**Citation:**
- [Sparkle Documentation: Delta Updates](https://sparkle-project.org/documentation/delta-updates/)
- [Sparkle PR #2668: Make binary delta version 4 the default](https://github.com/sparkle-project/Sparkle/pull/2668)
- [Sparkle issue #799: BinaryDelta performance on large binary files](https://github.com/sparkle-project/Sparkle/issues/799)

### 22. WKWebView Content Process Signing & Entitlements

**What goes wrong:** Concern: WKWebView uses a separate XPC content process (`com.apple.WebKit.WebContent`). Does this need separate signing or special entitlements?

**Why (MEDIUM confidence):** For **sandboxed** apps, WKWebView requires `com.apple.security.network.client` entitlement *even for loading local HTML strings* — this is a documented WebKit defect. For **unsandboxed** apps (MDViewer's case), entitlements are largely meaningless: "Entitlements are only meaningful in Mac App Store (and hence sandboxed) apps."

**Mitigation:** None needed for MDViewer. The WebKit content process is signed by Apple as part of macOS, not by your developer cert. The `WKWebView.evaluateJavaScript` and `loadHTMLString` calls work with no extra entitlements when the app is unsandboxed. Don't add `com.apple.security.network.client` "just in case" — it's a sandbox concept.

If MDViewer ever moves to App Store distribution (sandboxed), you'd need:
- `com.apple.security.network.client` (for WKWebView, even with no actual network)
- `com.apple.security.app-sandbox`
- And then the full Sparkle sandboxing dance (XPC services, mach-lookup exceptions)

### 23. Sparkle Admin Authentication Prompt (Unsandboxed App)

**What goes wrong:** First Sparkle update prompts user with "MDViewer wants to install an update — Touch ID / admin password". Some users panic.

**Why:** When Sparkle installs an update over `/Applications/MDViewer.app`, it requires write access. If the user installed the app with admin privileges (typical for `/Applications/`), Sparkle uses an authorization service for the install. This shows a system-style admin prompt the first time.

**Mitigation:**
- Document this in release notes ("First update installation will request admin authentication — this is normal").
- Mostly unavoidable; it's a macOS security model feature.
- The app's Info.plist `LSUIElement` and other flags don't change this behavior.
- The prompt is reduced if the user installed the app to `~/Applications/` or any user-writable location, but conventional macOS apps live in `/Applications/`.
- Sparkle 2.x does NOT trigger TCC-style consent prompts for "MDViewer wants to install an update" — that's a different system. The admin authorization prompt is a standard SecurityAgent dialog for filesystem writes.

**Citation:**
- [Sparkle Documentation (general)](https://sparkle-project.org/documentation/)
- [no source found] — specific 13.0+ behavior change for admin prompt patterns; appears to be unchanged from prior releases.

### 24. Hardened Runtime Required for Notarization

**What goes wrong:** Build, sign without `--options runtime`, submit to notary → rejected with "The executable does not have the hardened runtime enabled."

**Why:** Hardened runtime is a strict notarization requirement (as of macOS 10.14.5+). Without it, the notary refuses outright.

**Mitigation:** Always include `--options runtime` in every codesign call:
```bash
codesign --sign $SIGNING_ID --options runtime --timestamp \
  --entitlements MDViewer.entitlements \
  MDViewer.app
```
For MDViewer (no entitlements file currently, just hardened runtime), the `--entitlements` flag is optional — but `--options runtime` is mandatory. Verify with:
```bash
codesign -dvvv MDViewer.app 2>&1 | grep "flags="
# Expected: flags=0x10000(runtime)
```

### 25. Timestamp Server Outages Cause Signing to Hang

**What goes wrong:** `codesign --timestamp` reaches out to Apple's timestamp authority (`timestamp.apple.com`). When that server is having issues, codesign hangs or fails with timestamp errors.

**Why:** `--timestamp` is required for notarization. Without it, signature lacks the secure timestamp. Some apps disable it for speed, but you can't notarize without it.

**Mitigation:**
- Don't disable `--timestamp`.
- Add a retry loop in CI for codesign calls (3 attempts with backoff).
- Apple's timestamp server has had occasional outages; check `developer.apple.com/system-status` if codesign fails repeatedly.
- [no source found] for specific timestamp server outage incident reports — the issue is real but rarely impactful enough to generate dedicated bug reports.

**Citation:**
- [Apple: Resolving common notarization issues](https://developer.apple.com/documentation/security/resolving-common-notarization-issues)

### 26. Notary Service Rate Limiting (Documented But Not Specified)

**What goes wrong:** High submission frequency from a single team triggers rate limiting. Apple documents that this exists in their "Customizing the notarization workflow" guide but does not publish exact thresholds.

**Why:** Anti-abuse measure for the notary service. Concerning for projects with many sub-builds notarizing in parallel.

**Mitigation:** For MDViewer's expected pace (one notarization per release tag, maybe a couple per month), this is a non-issue. Becomes relevant if you start notarizing every PR or every nightly build — don't.
